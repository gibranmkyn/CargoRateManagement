HMW-37: FM Trucking Rate Sheet

How might we design the FM Trucking rate card to match the real-world pricing structure — a route × truck type matrix — separate from the L2 fee-based rate sheets used by other services?
What the real rate cards look like (from Hale-sun & Gonda Excel sheets):

FM Trucking (FTL) pricing is fundamentally different from other services:
  • Rows = Origin → Destination routes, often with district-level granularity (Guangzhou-Conghua, Guangzhou-Fangcun, etc.)
  • Columns = Truck types by tonnage: 1.5T, 3T, 5T, 8T, 10T, 12T, 40HQ, 45HQ
  • Each truck type has capacity specs: weight cap (kg), volume cap (CBM), and dimensions
  • Rate = flat per trip for that truck type on that route
  • Not all truck types available on all routes (some cells empty)
  • Currency varies: HKD for cross-border (China→HK), RMB for domestic China routes

Contrast with EC/CS/CR/OH: These use the L2 Cost ID structure (fee type × location). FM doesn't have L2 sub-fees in the same way — the "sub-fee" dimension is truck type, not fee category.

Option A: Split Rate Page — FM Tab + Services Tab

Recommended
Thesis: The Rates page gets two top-level tabs: Trucking and Services. Trucking shows the route × truck type matrix (FM-specific). Services shows the L2 fee × location grid (EC, CS, CR, OH). Each tab has the vendor selector. This reflects the fundamental structural difference in how these rates are priced.

FM Trucking tab:

  • Rows = routes (origin → destination), grouped by origin zone
  • Columns = truck types (1.5T, 3T, 5T, 8T, 10T, 12T, 40HQ, 45HQ)
  • Column sub-headers show capacity specs: "< 2500kg / 18CBM"
  • Each cell = flat rate per trip. Currency shown per row (route-level)
  • Inline editable, tab between cells
  • "+ Add route" to add new rows

Services tab: Same as current L2 fee × location/route grid for EC/CS/CR/OH.

✓ Pros
• Matches actual rate card format from Excel sheets
• Truck type columns are intuitive for trucking ops
• Clean separation — FM and other services don't compete
• Can add truck type specs (weight/volume caps) in column headers
• District-level route granularity supported naturally
✗ Cons
• Two tabs = two mental models for managing rates
• "Services" tab name might confuse (EC is also a "service")

Option B: Unified Grid — FM Truck Types as L2 Rows

Force-Fit
Thesis: Treat each truck type as an L2 sub-service: FM-1.5T, FM-3T, FM-5T, FM-8T, FM-10T, FM-40HQ. Then the existing L2 × route grid works — FM rows are truck types, columns are routes, same as EC/CS. One unified rate sheet for everything.
✓ Pros
• One rate sheet design for all services
• No need for a separate FM tab
• Simpler to build — reuses existing grid
✗ Cons
• Conceptually wrong — a truck type is NOT a fee category
• Loses truck specs in headers (weight/volume caps)
• Column headers become awkward — route columns mixed with L2 fee rows
• When ops creates a job, they pick ONE truck type, not multiple
• Breaks the mental model: in real rate cards, truck types are COLUMNS
⚠ This option forces truck types into the L2 row structure where they don't belong

Verdict: Option A — Split Rate Page with FM Tab

Recommended because: FM Trucking pricing is structurally a different beast. The rate dimension is truck type, not fee category. Forcing truck types into L2 rows (Option B) breaks the mental model — ops picks ONE truck type per trip, not multiple fees. The split tab approach mirrors reality: trucking rate cards are route × truck type matrices, and that's exactly what the FM tab shows.

  • Rates page tabs: 🚛 Trucking (FM) | 📦 Services (EC, CS, CR, OH)
  • FM tab grid: Rows = routes (grouped by zone). Columns = truck types (1.5T through 45HQ)
  • Truck type specs in column sub-headers: weight cap, volume cap
  • Currency per route: RMB for domestic, HKD for cross-border
  • Zone grouping + collapse for scalability (same pattern as HMW-32)
  • Create form impact: When creating an FM job, ops also selects a truck type → system looks up the rate for that route + truck type combination
  • Data model: FM rates use truckType field instead of costId. Or reuse costId as "FM-3T", "FM-5T" etc.
  • Accessorials (FM001 Warehouse Transfer, FM002 Pickup Fee, FM003 Cross-Border) become optional add-on fees below the main trucking rate — not the primary pricing dimension
