HMW-44: Job-Level View

"How might we give ops planners a job-level view that enables vendor coordination, proof chasing, and exception handling — without duplicating the delivery order view?"
Why this matters: The current Delivery Orders view groups everything by client request. But ops planners spend most of their day thinking about vendor execution: "Which jobs need trucks today? Who hasn't uploaded proof? What got rejected?" These questions require mentally flattening jobs across 15+ DOs — impossible to answer from the current view without expanding every row.

Key insight: DO view = demand-side (client requests). Job view = supply-side (vendor execution). Complementary, not redundant.

Option A: Power Table

Recommended
Thesis: Flatten all jobs into one dense table with powerful filters. The "better Excel" approach — ops builds their own views by combining service, vendor, proof status, and date filters. Maximum flexibility, minimum new UI concepts.

How it works: Every job across all delivery orders appears as one row. Service type pills let ops filter to just FM or just EC. Proof status pills highlight what needs chasing. DO reference is a compact clickable link — enough context without dominating.
Pros
• Zero learning curve — it's a table with filters, same pattern as DO view
• Answers ANY job question by combining filters (FM + Awaiting + Today = morning dispatch)
• Dense — 30+ jobs visible without scrolling
• Scales cleanly to 500+ jobs with pagination
Cons
• Requires ops to build their own filter combos — no pre-built workflows
• Vendor workload not visible at a glance (must filter to one vendor)
• Could feel like "the same thing but without the DO grouping"

Option B: Smart Views

Opinionated
Thesis: We know the 4 questions ops planners ask every day. Pre-build a view for each one instead of making them construct filters from scratch. "Today's Dispatch" shows FM jobs due today. "Proof Chasing" groups jobs by vendor so you know who to call. "Exceptions" surfaces rejected jobs and cost issues. "All Jobs" is the power-table fallback.

How it works: Horizontal view tabs replace the filter bar. Each tab pre-configures the columns, sorting, grouping, and filters for its use case. The underlying data is the same — the view just changes the lens.
Pros
• Zero friction — click a tab and immediately see what you need
• "Proof Chasing" groups by vendor = instant "who do I call?" answer
• Exception count badge creates urgency (red "2" on the tab)
• Teaches new ops planners how to think about their workflow
Cons
• Rigid — what if ops needs a view we didn't pre-build?
• 4 views to maintain (each is essentially a different page)
• "Today's Dispatch" only useful for FM — other services don't dispatch by hour
• Grouping adds visual complexity vs a flat table

Option C: Vendor Workboard

Focused
Thesis: Jobs are inherently vendor-centric — every job is a vendor doing work. Put the vendor list on the left as a permanent sidebar with workload summaries. Click a vendor, see their jobs on the right. The sidebar gives you the "who needs attention?" answer instantly.

How it works: Left sidebar shows each vendor with colored dots representing their jobs (green = validated, blue = uploaded, gray = awaiting, red = rejected). Click a vendor to see their full job table on the right, with a summary bar showing totals. Like an email client — vendors on the left, jobs on the right.
Pros
• Vendor workload visible instantly — colored dots show who's behind
• Natural for proof chasing: "HaleSun has 3 grays, let me click and see"
• Clean separation — sidebar for overview, table for detail
• Feels like a purpose-built vendor management tool
Cons
• Can only see one vendor at a time — can't compare across vendors
• Sidebar takes horizontal space in an already dense layout
• New UI pattern not used elsewhere in the app (sidebar + table)
• "All vendors" mode would just be Option A with less space

Recommendation: Option A (Power Table) with cherry-picks from B

Start with the flat Power Table because it follows the same pattern as the existing DO view — zero learning curve, maximum flexibility, and it scales to any question ops can ask. The filter bar (service pills + proof status pills + vendor dropdown + date) lets ops build exactly the view they need for morning dispatch, proof chasing, or exception handling.

Cherry-pick from Option B: Add a "group by" dropdown (None / Vendor / Service / Date) to the filter bar. This gives ops the ability to optionally group the flat table — getting the proof-chasing-by-vendor layout from Option B when they need it, without baking it in as a separate "view" that needs maintenance.

Skip Option C — the vendor sidebar is visually appealing but constrains ops to one vendor at a time. The filter-based approach is strictly more flexible, and the sidebar introduces a new layout pattern not used elsewhere.

The key interaction: Clicking a job row opens the same slide-out panel from the DO view (proof upload, fee management, activity log). Clicking the DO link navigates to the DO view with that order expanded. This makes the two views deeply connected rather than siloed.
