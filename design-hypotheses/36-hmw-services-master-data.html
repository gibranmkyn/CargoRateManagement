HMW-36: Services Master Data — L1/L2 Hierarchy

How should the Services tab in Master Data display and manage the L1/L2 service hierarchy with Cost IDs?
Current state: The Services tab in Master Data is a "Coming soon" placeholder. But the L1/L2 service taxonomy is the backbone of the rate management system — it defines the 20 Cost IDs across 5 L1 services that generate rate sheet rows and auto-populate fees on jobs.

The taxonomy: 5 L1 services (CR, CS, EC, FM, OH) contain 19 L2 sub-services total — each with a Cost ID (e.g., FM001), a name, and a default unit type (/trip, /kg, /bag, /CBM). This is the source of truth that determines what rows appear in vendor rate sheets (HMW-31) and what fees auto-populate on jobs (HMW-25/26).

Why it matters now: Without a proper Services master data UI, admins can't see what fee types exist, can't add new ones, and have no visibility into which vendors have rates configured for each fee. The rate sheet (HMW-31) needs this as its row source.

Option A: Expandable Tree Table

Recommended
Thesis: The L1/L2 hierarchy IS a tree — the UI should reflect it. Each L1 service is an expandable row that reveals its L2 sub-services when clicked. Collapsed rows show a count of L2 fees. Expanded rows show the full detail. Color-coded L1 badges match the existing service tag styling. Columns show Cost ID, Name, Unit Type, Vendors Using, and Rates Configured — giving admins a full picture of the taxonomy's "coverage."

How it works:

  • Table with 5 L1 rows — click any to expand and see nested L2 sub-services
  • L2 rows show Cost ID (mono), name, unit type badge, vendor count, rates count
  • "+ Add L2 Fee" button within each expanded L1 section
  • Summary stats bar: total L1 services, total L2 fees, total rates configured
  • L1 badges use the existing service pill style (Future Blue, 99px radius)
Pros
+ Tree structure matches the mental model of L1/L2 hierarchy
+ Expand what you need, collapse what you don't — scales well
+ Vendor/rate counts give immediate visibility into coverage gaps
+ "Add L2 Fee" scoped to the right L1 parent — prevents misclassification
Cons
- Slightly more interaction to see all 19 fees (must expand each L1)
- Nested table indentation eats horizontal space
- Expand/collapse state not persisted across page navigations

Option B: Two-Panel Master-Detail Layout

Classic
Thesis: Left panel shows the 5 L1 services as a clickable list. Selecting one loads the right panel with that L1's L2 sub-services. Classic master-detail pattern used in settings/config pages. Clean separation of hierarchy levels.
Pros
+ Clean separation — L1 list always visible on the left
+ Right panel has full width for L2 detail (more room for columns)
+ Familiar pattern — settings pages, email clients, etc.
+ Easy to add L1-level metadata (description, rate type) to the right panel header
Cons
- Can't see all services at once — must click through each L1
- More clicks to get a holistic view of the full taxonomy
- Left panel wastes space with only 5 items in the list
- Two-panel layout feels heavy for such a small dataset

Option C: Flat Table With L1 Group Headers

Simple
Thesis: Show all 19 L2 fees in one flat table, grouped by L1 service header rows. No expand/collapse — everything visible at once. Similar to how the rates page currently groups rates by vendor. Header rows are visually distinct with the L1 badge and service name.

The dataset is small enough (19 rows) that nesting isn't strictly needed. A flat table with group headers provides full visibility with zero interaction cost.

Pros
+ Everything visible at once — zero clicks needed
+ Simple implementation — standard table with group header rows
+ Easy to scan, search, and sort
+ 19 rows fits on one screen — no scrolling for most monitors
Cons
- Doesn't visually communicate the L1/L2 hierarchy as strongly
- Group headers take up row space without data columns
- "Add L2 Fee" placement is less obvious (bottom of each group?)
- If taxonomy grows beyond 30+ fees, the flat list becomes unwieldy

Verdict: Option A — Expandable Tree Table

The L1/L2 hierarchy IS a tree. The UI should reflect that structure. Expand what you need, collapse what you don't. Each collapsed L1 row shows a count badge ("6 sub-services") so you always know the scope. Expanded rows show full L2 detail with Cost IDs in JetBrains Mono, color-coded unit type badges, and vendor/rate coverage counts.

  • Tree table — 5 expandable L1 rows, 19 L2 child rows total
  • Collapsed state — shows L1 badge, name, sub-service count, aggregate vendor/rate counts
  • Expanded state — shows L2 rows with Cost ID (mono), name, unit badge, per-fee vendor/rate counts
  • Unit badges — color-coded per existing design system: gray /trip, blue /kg, purple /bag, amber /CBM
  • "+ Add L2 Fee" button — scoped to each L1 section, prevents misclassification
  • Stats bar — 5 L1 services, 19 L2 fees, N rates configured, N active vendors
  • Source of truth — this table defines the rows that appear in vendor rate sheets (HMW-31) and the fees that auto-populate on jobs (HMW-25/26)

Why not Option B? Two-panel layout is overkill for 5 L1 items — left panel wastes 220px showing a 5-item list. Can't see the full taxonomy at once.
Why not Option C? Flat table works but loses the hierarchy affordance. At 19 rows it's fine, but it doesn't communicate the L1/L2 relationship as effectively. The tree table makes the parent-child structure explicit and allows progressive disclosure.
