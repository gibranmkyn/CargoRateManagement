HMW-29: Scalable Filtering for 1000+ Orders

How might we keep the delivery orders page fast and focused when the order count grows from 10 to 1000+, without losing the ability to find any historical order?
The scale problem: Today there are 10 seed orders and the Active filter works great. But in production, ops creates 10-50 orders/day. After 3 months: 1,000-4,500 orders. Even the "Active" filter will show 20-50 orders (manageable), but the "All" view becomes unusable. The "Completed" filter shows 900+ orders — no one scrolls through that. The question is: how do we make historical lookup fast without cluttering the daily view?

Research findings: Best-in-class TMS tools (Turvo, GoFreight, Descartes) use date range pickers as the primary historical filter. Baymard Institute recommends applied filters with visual indicators and "Clear All" for data-heavy tables. Virtual scrolling is recommended over pagination for operations tools where scanning speed matters.

Option A: Date Period Picker + Active Default

Recommended
Thesis: Keep the existing Active/All/Completed chips, but add a date period selector that applies when viewing "All" or "Completed." Active view has no date constraint (you always see everything active). "All" and "Completed" default to "This month" and offer quick presets: Today, This week, This month, Last month, Custom range.

How it works:

  • Active tab: No date filter needed — shows all active orders regardless of when created. This is always small (20-50 orders max).
  • All / Completed tabs: Date picker appears. Defaults to "This month." Quick presets for common lookups. Custom range for billing periods.
  • Pagination: Show 50 orders per page with prev/next. No infinite scroll (bad for data tables per research).
  • Sort: Default newest first (most recent pickup date). Sortable columns for ops to reorder.
✓ Pros
• Active view stays fast regardless of total order count
• Date picker is the natural historical lookup — "find Shopee orders from January"
• Quick presets match real workflows: "what shipped this week?"
• Pagination keeps rendering fast at any scale
• No backend changes needed for localStorage (filter client-side)
✗ Cons
• Date picker adds UI complexity to the filter bar
• "This month" default may miss orders from yesterday that span month boundaries

Option B: Month Tabs — Calendar as Primary Navigation

Calendar-First
Thesis: Replace the Active/All/Completed chips with month tabs at the top: Active | Mar 2026 | Feb 2026 | Jan 2026 | ... Active is always first. Historical months are automatically generated from order dates. Each month tab shows all orders from that period. This aligns with billing cycles (monthly invoicing).
✓ Pros
• Natural for billing: "pull up January's orders"
• Months auto-generate — no date picker needed
• Each month has a bounded number of orders (~150-500)
• Aligns with accounting periods
✗ Cons
• Tab bar grows long — after 12 months, 12+ tabs
• Hard to find orders that span month boundaries
• "Active" orders from different months mixed in the Active tab
• Loses the "All" view entirely

Option C: Search-First — MAWB/Customer Search as Primary

Search-Driven
Thesis: Active orders are always shown by default (same as now). For historical lookup, the primary tool is a prominent search bar — type a MAWB number, customer name, or order ID and get instant results across ALL time. No date filters needed if search is fast. This matches how ops actually looks up old orders: "find the Shopee order from that MAWB number."
✓ Pros
• Fastest path to finding a specific order
• No date guessing — MAWB search is exact
• Simplest UI — no date pickers, no month tabs
• Active view is untouched and always focused
✗ Cons
• Can't browse — only works if you know what you're looking for
• "Show me all Shopee orders from January" needs additional filters
• Doesn't help with billing review (need date ranges for that)
• Search quality depends on implementation

Verdict: Option A — Date Period Picker + Active Default

Recommended because: It handles both daily ops (Active view, no date filter) and historical review (date presets for billing/audit) in one unified filter bar. The date picker only appears when you leave the Active view, so it adds zero complexity to the daily workflow. Pagination at 50/page keeps rendering fast at any scale.

  • Active tab: No date constraint, no pagination needed (always <50 orders)
  • All/Completed tabs: Date period selector with presets (Today, This week, This month, Last month, Custom)
  • Pagination: 50 orders/page with page numbers. Not infinite scroll (bad for data tables).
  • Default period: "This month" when switching to All/Completed
  • Subtitle: "23 active · 1,247 total" gives context without cluttering
  • Future-proof: When backend is added, date filter maps directly to database query

Note: For localStorage (Phase 1), client-side filtering + pagination is sufficient up to ~5,000 orders. Beyond that, a backend with server-side pagination is required. The UI pattern here is designed to work with both.
