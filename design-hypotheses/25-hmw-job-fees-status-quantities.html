HMW-25: Multi-Fee Jobs, Status & Editable Quantities

How might we show that each job has multiple fee line items (L2 cost IDs), let ops planners edit quantities until a job is complete, and make status changes faster — so the billing total reflects reality and daily workflow feels effortless?
Why this matters: Currently, each job shows a single flat cost. But in reality, a trucking job might have 3 charges: base rate (RM 450), fuel surcharge (RM 45), toll fees (RM 30) = total RM 525. Ops planners need to see this breakdown, edit quantities (bags, weight) as they become known, and the billing total must reflect the sum of all fees. Additionally, the current status picker is a full 5-row vertical list in the slide-out — slow for the most common action (marking a job "In Progress" or "Completed").

Option A: Inline Fee Table + Quick Status Buttons

Recommended
Thesis: Each job row in the sub-table shows the total cost (sum of all fees). Expanding a job reveals the fee breakdown as an L2 table. Status changes happen via inline button chips on the sub-table row itself — no need to open the slide-out for the most common action. Quantities are editable inline until the job is Completed.
✓ Pros
• Status change in 1 click (no slide-out needed)
• Fee breakdown visible without leaving the page
• Quantities editable right where costs are shown
• Sub-table total is always the sum of visible fees
✗ Cons
• Sub-table gets 3 levels deep (order → jobs → fees)
• Inline editing adds complexity to the table
• More visual density in an already dense UI

Option B: Slide-Out Focus + Progress Bar Status

Current Pattern Enhanced
Thesis: Keep the sub-table minimal (total cost only, no inline status buttons). Status changes stay in the slide-out panel but use a compact progress-bar style instead of the current 5-row vertical list. Fee breakdown lives exclusively in the slide-out. Quantities are editable in the slide-out.
✓ Pros
• Sub-table stays simple and scannable
• Consistent UX — actions always in slide-out
• Progress bar gives clear visual of job lifecycle
• Less risk of accidental status changes
✗ Cons
• 2 clicks for most common action (open slide-out → change status)
• Fee breakdown hidden behind a click
• No at-a-glance status comparison across jobs

Option C: Hybrid — Inline Status + Expandable Fee Panel

Compromise
Thesis: Status buttons are inline on the sub-table (like Option A), but the fee breakdown opens in the slide-out (like Option B). Click a job row → slide-out opens with fees + quantities. The sub-table shows Total Cost but no L2 detail. Best of both worlds, but the slide-out becomes required for fee editing.
✓ Pros
• Fast status changes (1 click inline)
• Sub-table stays at 2 levels (order → job), not 3
• Fee detail in the slide-out is familiar
• Lower visual complexity in the main table

• Inconsistent: some actions inline, some in panel
• Can't see fee breakdown and other jobs simultaneously
• Slide-out gets crowded with fees + quantities + proofs + activity

Verdict: Option A — Inline Fee Table + Quick Status Buttons

Recommended because: The two most frequent ops actions are (1) changing job status and (2) checking costs. Option A makes both zero-click actions — status is visible and changeable directly on the sub-table row, and fees are visible by expanding the job row. The 3-level depth (order → job → fees) mirrors the actual data model (1 order has N jobs, 1 job has N fees), which ops planners coming from Excel will find natural.

Key design decisions:

  • Quick status: 3 primary buttons inline (Pending / In Prog / Done). Reject + Cancel in slide-out only (rare actions, need confirmation).
  • Fee breakdown: Expandable L2 table under each job row. Shows Fee Name, Rate × Qty = Amount.
  • Editable quantities: Bags, Weight, Volume editable in slide-out until job status = Completed. Changes recalculate all per-unit fees instantly.
  • Total Cost: Sub-table column shows sum of all fee line items. Order total shows sum of all jobs.
  • "+ Add fee" button: In the slide-out, ops can add custom fee line items (e.g., waiting time, special handling).
