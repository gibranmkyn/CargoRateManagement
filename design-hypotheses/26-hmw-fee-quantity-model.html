HMW-26: Fee Line Items & Quantity Model

How might we structure fee line items and quantities so that each job's cost accurately reflects multiple charges, quantities can differ per job, and the data flows correctly into billing?
The real scenario: TikTok ships 24 bags / 1,240 kg. FM Trucking handles all 24 bags. But at customs, 4 bags are held for inspection — only 20 bags clear. The EC job should calculate customs fees on 20 bags, not 24. Meanwhile, the FM job has 3 fee items: base rate (flat), fuel surcharge (per-km), and toll fees (flat). The quantity multiplier differs per fee type — bags for per-bag rates, weight for per-kg rates, distance for per-km rates. How do we model this without overwhelming the ops planner?

Option A: Order-Level Quantities + Per-Fee Override

Recommended
Thesis: Bags and weight live at the order level (where they belong logistically — you ship 24 bags total). Each job inherits order quantities as default. But each fee line item has its own quantity field that CAN be overridden. The fee's rate unit determines what the quantity means: bags for /bag, kg for /kg, trips for /trip.

Data model:

Job {
  fees: FeeLineItem[]     // 1..N fee line items per job
}

FeeLineItem {
  id: string
  name: string            // "Base rate", "Fuel surcharge", "Toll fees"
  rateId?: string         // linked to VendorRate if auto-populated
  currency: Currency
  rate: number            // unit price
  unit: RateUnit          // flat, per-kg, per-bag, per-cbm, per-km
  quantity: number        // defaults from order, overridable per fee
  amount: number          // = rate × quantity (calculated)
}
Fee creation flow:
  1. Auto-populated from rate card: When vendor + service + location are selected, the system creates a FeeLineItem from the matched VendorRate. Quantity auto-fills from the order (bags for /bag, weight for /kg, 1 for /trip).
  2. "+ Add fee" for surcharges: Opens an inline form with: Name (dropdown of common fees OR free text), Rate amount + currency, Unit type, Quantity (defaults from order). For a fee catalog, use a small predefined list: fuel surcharge, toll fees, waiting time, special handling, overtime, insurance.
  3. Manual override: Ops can edit any fee's quantity (e.g., change bags from 24 to 20 for customs). The amount recalculates immediately.
✓ Pros
• Order-level bags/weight stays as the source of truth
• Per-fee override handles edge cases (partial shipments)
• Fee catalog keeps surcharge names consistent
• Amount always = rate × quantity (transparent, auditable)
✗ Cons
• More fields per fee (rate + unit + qty + amount)
• Ops must know to override qty for partial shipments
• Fee catalog needs maintenance as new fee types emerge

Option B: Per-Job Independent Quantities

Maximum Flexibility
Thesis: Each job has its own bags/weight/volume fields, independent from the order. When a job is created, it inherits from the order, but after that it's independent. Fee quantities are derived from the job's quantities (not the order's). This means each job tracks exactly how much cargo it handled.
✓ Pros
• Each job accurately reflects actual cargo handled
• Partial shipments naturally supported
• Billing is per-job accurate — no overcharging
✗ Cons
• More fields to fill per job (3 qty fields × N jobs)
• Order-level bags/weight becomes decorative
• Ops must update quantities on every job individually
• Inconsistency: order says 24 bags, sum of jobs might say 20+24+24

Option C: Simple — Order Quantities Only, No Override

Minimum Viable
Thesis: Keep it simple. Order-level bags/weight apply to ALL jobs. Each fee's quantity is calculated from the order's quantities based on the fee's unit type. No per-job or per-fee overrides. If a partial shipment happens, ops adjusts the order quantities.
✓ Pros
• Simplest model — one place to edit quantities
• No confusion about which quantity is "real"
• Fastest to implement
✗ Cons
• Can't handle partial shipments per job
• Changing order qty affects ALL jobs retroactively
• Not accurate for billing if different jobs handle different amounts

Verdict: Option A — Order-Level Qty + Per-Fee Override

Recommended because: It gets the default right (order quantities flow down to all fees) while allowing surgical overrides when reality differs. The fee catalog (fuel, tolls, waiting time, special handling, overtime, insurance) keeps data clean. The "+ Add fee" inline form is small and focused. This model handles 95% of cases automatically and the 5% edge cases with a single quantity edit.

  • Quantities: Live at order level. Each FeeLineItem has a qty field that defaults from order but can be overridden.
  • Auto-populate: Rate card match → creates base FeeLineItem automatically. Qty determined by unit type (/bag → bags count, /kg → weight, /trip → 1).
  • Manual fees: "+ Add fee" offers a dropdown of 6 common surcharges + free text for custom. Each needs: name, rate, unit, qty.
  • Lock on complete: When job status = Completed, all fees and quantities become read-only. Edit requires changing status back.
  • Fee catalog: Fuel surcharge, Toll fees, Waiting time, Special handling, Overtime, Insurance. Hardcoded initially, configurable in Phase 3.
