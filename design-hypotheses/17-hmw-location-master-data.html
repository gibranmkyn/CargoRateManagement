HMW: Location Master Data

"How might we structure locations as managed master data — so rate lookups, route matching, and reporting all work reliably instead of depending on free text?"
Current problem: Ops types locations as free text — "TikTok WH, Shenzhen" vs "TikTok Warehouse, Shenzhen" vs "tiktok wh shenzhen". These are treated as three different locations by the system.

Impact: Rate cards need exact location matches to work. If the rate is set for "TikTok WH, Shenzhen" but ops types "TikTok Warehouse Shenzhen", the lookup returns nothing. Reporting groups by exact string, so shipment volume at the same warehouse appears as three separate entries. This is the #1 prerequisite for rate management — without consistent location IDs, rate lookups break silently.

Option A: Location Dropdown in Job Form

Strict control
How it works: Replace free-text origin/destination fields with dropdown selects populated from a managed locations master list. Each location has a code (LOC-001), display name, zone, and type (warehouse/port/airport/checkpoint/hub). Only admin can add new locations. Ops always picks from the list.
Pros
- Perfect consistency — every job references a canonical location ID
- Rate lookup works 100% of the time (exact match on location code)
- Clean reporting — no duplicates, grouping always correct
- Simple UI — just a dropdown, no fuzzy logic needed
Cons
- Ops can't enter a new location without admin adding it first
- Could slow down order creation if a new warehouse appears
- Ops must wait or message admin to unblock themselves
- If the list grows large, dropdowns become hard to scan

Option B: Autocomplete with Fuzzy Matching

Flexible input
How it works: Keep text inputs for origin/destination, but add autocomplete powered by the locations master list. Typing "tiktok" shows "TikTok WH, Shenzhen" as the top match. Typing "yant" shows "Yantian Port". If no match is found, ops can type a new name and it gets created as a new location in master data (with a confirmation prompt to set zone and type).
Pros
- Fast for both known and new locations
- Doesn't block ops when a new location appears
- Fuzzy matching catches misspellings and abbreviations
- Feels like typing, not picking from a list
Cons
- Still possible to create duplicates ("TikTok WH" and "TikTok Warehouse")
- Needs dedup logic or manual cleanup by admin
- Fuzzy matching quality depends on the algorithm
- New locations may be created with incomplete metadata

Option C: Dropdown + "Add New" Option

Recommended
How it works: Dropdown like Option A (strict, from master list), but with an "Add new location" option at the bottom of every dropdown. Clicking it opens a small inline form right below the dropdown where ops enters: name, zone, and type. The new location is immediately available for selection and gets added to master data. Admin can review and clean up later.
Pros
- Consistency by default — 99% of the time ops picks from the list
- Flexibility when needed — new locations don't block ops
- New locations enter the system with proper metadata (zone, type)
- Rate lookup works reliably, just like Option A
Cons
- Slightly more complex UI than a plain dropdown
- Ops could still create near-duplicates ("Yantian" vs "Yantian Port")
- Admin needs a way to see recently-added locations and merge duplicates

Recommendation: Option C — Dropdown + "Add New"

Why Option C: The dropdown enforces consistency by default — 99% of jobs use existing locations, so rate lookups and reporting work reliably. The "Add new" escape hatch prevents ops from being blocked when a genuinely new location appears (new warehouse, new port terminal). New locations enter the system with proper metadata (name, zone, type) from day one.

Admin review workflow: New locations created by ops are flagged as "Pending review" in the Master Data page. Admin can approve, edit, or merge with an existing location. This catches near-duplicates without slowing down ops.

Data model: Add a Location entity: { code, name, zone, type, status: "active" | "pending_review" }. Replace free-text origin/destination in Jobs with location code references. Seed with 12 known locations. This is the foundation for rate management — build this first.
