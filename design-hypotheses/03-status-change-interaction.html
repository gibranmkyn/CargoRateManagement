H04: Status Change Interaction

Ops planners change job statuses ~50 times/day. This is the single most repeated action in the app. The interaction must be fast (< 2 clicks), provide enough context to avoid mistakes, and leave an audit trail.

Key constraint: Status changes are non-sequential (customs can complete before trucking). The UI must NOT imply a linear progression — just show the current state and let the user pick any new state.

Option A: Inline Dropdown on Job Chip

Fastest — 1 click to open, 1 click to select
How it works: Click the status chip directly in the trip table row. A dropdown appears anchored to the chip showing all 5 statuses. Select one — done. The chip updates instantly. Activity log entry is created automatically in the background.
✓ Pros
• Fastest possible interaction (2 clicks total)
• No navigation, no context switch
• Works for bulk status updates (click, click, click down the list)
• Mirrors how spreadsheet users expect to update a cell
✕ Cons
• No room for rejection reason (would need a follow-up prompt)
• Can't upload proof in the same flow
• Easy to misclick — no confirmation
• Small touch target on mobile

Option B: Slide-Out Panel

Best balance — fast + contextual
How it works: Click a job chip or the "J01→" link. A panel slides in from the right showing: status picker, job details (vendor, route, services), proof upload, and recent activity log. The trip list stays visible on the left. Change status, upload proof, or view log — all without navigating away.
✓ Pros
• Fast — no page navigation
• Full context: status, route, services, vendor, proofs, log all visible
• Can upload proof + change status in the same flow
• Trip list stays visible for reference
• Rejection reason can be entered inline
✕ Cons
• More complex to implement (panel animation, overlay handling)
• Reduces visible trip list width when open
• May feel cramped on smaller screens

Option C: Navigate to Job Detail Page

Most context — but slowest
How it works: Click job chip → navigates to the full job detail page (what we have now). Change status there, upload proofs, view full activity log. Hit back to return to trip list.
✓ Pros
• Maximum context — full screen for the job
• Best for complex actions (multiple proof uploads, detailed log review)
• Simplest to implement
• URL-addressable — can share link to a specific job
✕ Cons
• Slowest — full page navigation + back button
• Loses trip list context (can't see other trips while working on one job)
• 50 status changes/day × 2 navigations each = 100 page loads
• Kills scanning flow

Recommendation: Option B (Slide-Out Panel)

The slide-out panel is the best balance for ops planners who change status 50x/day:

  • Fast enough: 1 click to open, 1 click to change status. Not as fast as inline dropdown (Option A) but close.
  • Enough context: Status picker, route, services, proof upload, and activity log — all visible without navigation. Prevents misclicks because the user can see WHAT they're changing.
  • Proof upload in flow: The killer advantage. When a vendor completes a job, the planner can change status to "Completed" AND upload the proof document in the same interaction — without navigating to a separate page.
  • Trip list stays visible: The dimmed trip list behind the panel provides context and allows quick switching between jobs.
  • Full detail as escape hatch: "Open full detail →" link at the bottom for cases that need the complete job detail page.

Option A (inline dropdown) is fastest but lacks proof upload and rejection reason context — it would need follow-up prompts that negate the speed advantage. Option C (navigate) is too slow for the volume of status changes ops does daily.
