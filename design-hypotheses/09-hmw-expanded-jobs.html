HMW: Expanded Job View

"How might we show jobs within an expanded delivery order in a way that feels native to Excel users — not like a gallery of cards?"

Option A: Sub-Table (most Excel-like)

Recommended
How it works: Expanding a delivery order reveals a nested table with one row per job. Same column alignment as the parent table but indented. Feels like expanding a grouped row in Excel.
✓ Pros
• Most natural for Excel users — it's just more rows
• Highest data density — 3 jobs in ~100px vs ~200px for cards
• Column alignment makes scanning easy
• Click any job row to open slide-out
✕ Cons
• Less visual distinction between parent and child rows
• Can't show as much detail per job as a card
• Route info might be truncated in narrow columns

Option B: Compact Cards (current approach, tightened)

Current
How it works: What we have now — parallel cards with vendor, services, route, status. Already tightened from the dense mockup. Each card is a self-contained job summary.
✓ Pros
• Visual separation between jobs is very clear
• Each job is a distinct visual unit
• Status-colored borders give quick scanning signal
✕ Cons
• Takes more vertical space than a sub-table
• Cards feel "designed" not "data" — friction with Excel users
• Information is arranged differently from the parent table

Option C: Inline Rows (no expand needed)

Radical
How it works: Don't expand at all. Each job IS a row in the main table, grouped under its delivery order. The delivery order header row is slightly styled (bold, slightly darker background) and the job rows are indented below it. Like grouped rows in Excel with a subtotal row.
✓ Pros
• Maximum density — no expand/collapse, everything visible
• Most Excel-like — it literally IS a grouped spreadsheet
• No context switching between "order view" and "job view"
• Fastest scanning — all info in one flat list
✕ Cons
• With 50 orders × 2-3 jobs each = 100-150 rows. Could be overwhelming.
• Harder to visually group "which jobs belong to which order"
• Loses the clean "5 orders" → expand for detail flow

Recommendation: Option A (Sub-Table)

Option A is the best fit for Excel-native users. It keeps the expand/collapse flow (so the list stays manageable at 50 orders) but the expanded content is a TABLE, not cards. Same column alignment, same visual language, same density.

Option C (inline rows) is the most radical and densest, but 50 orders × 2-3 jobs = 150 rows is too much for a single flat view. It would need virtual scrolling and grouping logic.

Option B (cards) should be dropped — it fights the dense table aesthetic we just established.

The sub-table also naturally shows the "Proofs" column — the admin can see at a glance which jobs have uploaded proof documents and which don't, without expanding further.
