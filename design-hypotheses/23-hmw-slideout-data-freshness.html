HMW: Slide-Out Data Freshness

"How might we keep the slide-out panel's data fresh when ops changes status or uploads proofs -- so they always see the current state without closing and reopening?"
Current bug: The slide-out panel receives a job object snapshot when it opens. When ops changes status (e.g., Pending → In Progress), the mutation goes to React Context → localStorage. But the panel holds the original snapshot. The status picker visually stays on "Pending" even though the job is now "In Progress" in the global state. Ops has to close and reopen the panel to see the updated state.

What triggers this: Any mutation dispatched from within the panel: status changes, proof document uploads, activity log notes, rejection → reassignment. The panel's local data gets stale immediately after the action completes. This affects every interaction in the slide-out.

Current code: Panel state is { trip: Trip, job: Job, jobIdx: number }. These are copies passed when the panel opens. Context mutations update the canonical trips[] array, but the panel doesn't re-read from it.

Option A: Read from Context (Reactive Panel)

Recommended
How it works: Instead of passing the full trip + job objects to the panel, pass only tripId + jobId. The panel derives the current job from context on every render: trips.find(t => t.id === tripId)?.jobs.find(j => j.id === jobId). When a status change dispatches to context, context updates → the panel re-renders with the fresh data. No local copy, no drift, no stale state.
Pros
- Always fresh -- single source of truth (context)
- No local state drift or sync logic
- Simplest mental model: panel always shows what context has
- Small refactor: change panel state from {trip,job} to {tripId,jobId}
- Works for ALL mutations, not just status changes
Cons
- Requires refactoring how the panel receives data (IDs instead of objects)
- The .find() lookup runs on every render (negligible perf impact for ~50 trips)
- If tripId/jobId somehow becomes invalid (deleted trip), panel shows nothing (needs guard)
- All panel components must be updated to use derived data

Code change: panel state refactor

TripContext.tsx -- panel state type
// BEFORE: panel holds data copies
type PanelState = {
  trip: Trip;       // snapshot at open time
  job: Job;        // snapshot at open time
  jobIdx: number;
} | null;

// AFTER: panel holds IDs, derives data from context
type PanelState = {
  tripId: string;    // stable reference
  jobId: string;     // stable reference
} | null;
SlideOutPanel.tsx -- deriving data
const { trips, panel } = useTripContext();

// Derive fresh data on every render
const trip = trips.find(t => t.id === panel.tripId);
const job = trip?.jobs.find(j => j.id === panel.jobId);

// Guard: panel references deleted data
if (!trip || !job) return null;

// Now job.status is always the current value from context
// When ops changes status, context updates, component re-renders,
// and job.status reflects the new value immediately.

Option B: Optimistic Update in Panel State

Pragmatic
How it works: Keep passing the job object to the panel. But when the panel dispatches a mutation (e.g., status change), it ALSO updates its own local copy of the job. The panel maintains a local patched version alongside the context update. Both update simultaneously: context gets the canonical update, panel gets an optimistic local patch. No refactoring of how the panel receives data.
Pros
- No refactoring of panel state shape -- keeps {trip, job, jobIdx}
- Panel updates instantly (optimistic)
- Works for all panel-initiated mutations
- Minimal code change: add a local setState after each dispatch
Cons
- Two sources of truth: panel local state + context state
- Risk of drift: if context rejects the update or formats data differently
- Every new mutation type needs a matching local state patch
- External changes (another tab, background sync) won't reach the panel
- Violates single-source-of-truth principle

Code pattern: dual update (context + local)

SlideOutPanel.tsx -- dual state update
// Panel holds a local copy of the job
const [localJob, setLocalJob] = useState(panel.job);

const handleStatusChange = (newStatus: string) => {
  // 1. Update context (canonical source)
  dispatch({ type: 'UPDATE_JOB_STATUS', payload: { ... } });

  // 2. Optimistic local patch
  setLocalJob(prev => ({
    ...prev,
    status: newStatus,
    activityLog: [...prev.activityLog, newEntry]
  }));
};

// Problem: localJob and context can drift
// e.g., context adds a timestamp, local copy doesn't know about it

Option C: Close-and-Reopen Pattern

Workaround
How it works: After any mutation dispatched from the panel (status change, proof upload), the system auto-closes the panel with a brief slide-out animation (150ms), shows a toast confirmation, then re-opens the panel (150ms slide-in) with fresh data pulled from context. The reopen uses the same tripId + jobId to fetch the now-updated data. Effectively automates the "close and reopen" that ops currently does manually.
Pros
- Guarantees fresh data -- panel always opens with current context state
- Clear visual feedback: the close/open animation signals "something changed"
- No refactoring of panel data model (still passes full objects)
- Toast confirmation happens during the brief close, so no wasted time
Cons
- Jarring UX: panel flickers closed and open on every status change
- Feels like a hack, not a feature
- 300ms animation delay adds up with multiple rapid changes
- Scroll position in the panel resets on reopen
- If ops is mid-typing a note when the panel re-opens, input is lost

Problem: the 300ms flicker
Every status change, proof upload, or note causes the panel to visibly close and reopen. For ops making 3 quick changes (upload proof + change status + add note), that's 3 flickers in rapid succession. Scroll position resets each time. Any in-progress typing is lost. It feels like the app is glitching, not working as designed.

Verdict: Option A -- Read from Context by ID

The reactive panel is the cleanest architecture. Change panel state from { trip, job, jobIdx } to { tripId, jobId }. The panel derives its data from context on every render: trips.find(t => t.id === tripId)?.jobs.find(j => j.id === jobId). Always fresh, no local state drift, no close/reopen hacks.

Why not Option B? Two sources of truth is an anti-pattern. The optimistic local patch works for status but drifts on timestamps, activity logs, and any field the reducer computes. Every new mutation type needs a matching local patch -- this doesn't scale. Why not Option C? The close-and-reopen flicker is a hack that trades UX quality for implementation simplicity. It resets scroll position, loses in-progress input, and feels broken during rapid multi-action sequences.

Refactor scope: Change the OPEN_PANEL action to pass IDs instead of objects. Update the panel component to derive trip/job from context. Add a null guard for deleted trips/jobs. Estimated: ~30 lines changed across TripContext + SlideOutPanel.
