HMW-30: Proof-Centric Jobs + Rate-Locked Fees

How might we simplify the job lifecycle so that job progress is driven by proof of service — not manual status clicks — and all fees come strictly from configured rate cards?
Two problems, one root cause:

1. Fee free-text breaks billing integrity. The current "+ Add fee" lets ops type any name and amount. But the whole point of rate cards is that rates are pre-agreed with vendors. If ops can add arbitrary fees, billing validation becomes meaningless — you're comparing invoice amounts against numbers ops made up, not contractual rates. Every fee should originate from a rate card entry.

2. Manual job status is redundant busywork. Ops currently clicks Pending → In Progress → Completed on every job, 50x/day. But what actually matters is: did the vendor do the work? The proof of service (POD photo, customs doc, X-ray receipt) is the answer. If proof is uploaded and validated → the job is done. If not → it's not. The status should derive from the proof state, not the other way around.

The new mental model:
• Fees = what the rate card says. No manual entry. If a fee isn't in the rate card, add it there first.
• Job progress = proof state: Awaiting proof → Proof uploaded → Validated (ready for payment).
• Order complete = all jobs validated.

  • Awaiting proof — job created, vendor assigned, no proof doc uploaded yet. This is the default state.
  • Proof uploaded — vendor or ops uploaded a proof document (POD photo, customs clearance PDF, X-ray receipt). Auto-transitions when first document is uploaded.
  • Validated — ops admin reviewed the proof and confirmed it's acceptable. One click: "Validate" button. This means the job is complete and ready for payment.
  • Disputed — proof was reviewed but rejected (wrong document, incomplete, unclear). Vendor must re-upload. Optional state, replaces "Rejected."

Key difference: No manual "In Progress" state. If the vendor is doing the work but hasn't uploaded proof yet, the status is still "Awaiting proof." The system doesn't need to know the vendor is mid-delivery — it only cares about the outcome (proof).

Fee model:

  • All fees auto-populated from rate cards. When a job is created (vendor + service + location selected), the system looks up ALL matching rates and creates fee line items automatically.
  • No free-text fee entry. The "+ Add fee" button is removed. If a fee needs to be added, go to Rates → add the rate card entry → it appears on future jobs.
  • Quantities are editable (bags, weight, etc.) — this recalculates the fee amounts from the locked rates.
  • Rate card is the single source of truth for what a vendor charges.
✓ Pros
• Eliminates manual status busywork (50 clicks/day saved)
• Billing integrity guaranteed — fees always match rate cards
• Proof is the actual deliverable, not an arbitrary status click
• Simpler mental model: upload proof → validate → pay
• "Active" = orders awaiting proof or validation (natural filter)
✗ Cons
• Loses granularity of "In Progress" — can't see if vendor started
• Vendor surcharges outside rate card need a separate flow
• Requires rate cards to be comprehensive before jobs work properly

How "Active" Changes with Proof-Centric Model

The Active filter now maps to proof state:
  • Active = order has at least 1 job NOT validated (awaiting proof OR uploaded but not validated)
  • Completed = ALL jobs in the order are validated
  • This is cleaner than the old model because "Completed" has an unambiguous meaning: proofs validated, ready for payment.

Verdict: Adopt Proof-Centric Model + Rate-Locked Fees

This is a significant simplification that aligns the software with the real workflow. Ops doesn't care whether a vendor is "in progress" — they care whether the proof document is in hand and verified. The manual status clicks are replaced by a natural progression: upload proof → validate → pay.

  • Job states: Awaiting proof → Proof uploaded → Validated → (Disputed if proof rejected)
  • Auto-transition: "Awaiting" → "Uploaded" happens automatically when a document is uploaded
  • Manual action: Only "Validate" and "Dispute" are manual clicks
  • Fees: All from rate cards. No free-text entry. "Fees are configured in Rates →" link instead of "+ Add fee"
  • Quantities: Still editable per job (recalculates fees from locked rates)
  • Active filter: "Active" = orders where any job is NOT validated
  • Billing integration: "Validated" jobs = ready for billing. Clean handoff.

What gets removed: Status picker (5 buttons), "+ Add fee" free-text form, fee catalog dropdown. What remains: Upload proof, Validate, Dispute, Quantity editing.
