HMW-51: Cancellation, Reassignment & Partial Completion

"How might we design the cancel/replace flow so that admins can efficiently reassign work while keeping a full, immutable audit trail visible to both admin and vendor?"
Three real scenarios drive this design:

A) Standard Cancel: Admin cancels HaleSun's CS job because customer changed plans. No replacement. Vendor sees "Cancelled" + reason.

B) Cancel & Replace (reassignment): Admin cancels Gonda's EC job (wrong customs license) and creates a new one for HaleSun. Current system overwrites Gonda's record — wrong. New model: cancel Gonda (with reason) + create new job for HaleSun (linked). Both vendors see their respective records.

C) Partial Completion (pickup not ready): FM truck arrives but goods aren't ready. Driver waited, fuel burned — vendor deserves partial payment. Mark as Completed with remark + adjusted fees. Then create follow-up job for retry. Two billable events, not one cancelled job.

Current flaw: Reassignment clears cancelReason and overwrites vendor on the same job. Vendor A loses their record. No linkage between old and new.

Option A: Cancel Action Menu + Inline Replacement

Recommended
Thesis: Add a "Cancel Job" button (red, secondary) to the slide-out Status Action Bar for Pending/In Progress jobs. Clicking it reveals an inline cancellation form: mandatory reason textarea + two actions: "Cancel Only" and "Cancel & Replace." Cancel & Replace expands a vendor picker inline — one flow, one panel, no modal. Cancelled jobs become immutable and show a link to the replacement job if one exists.
Pros
• Everything happens in the slide-out — no navigation, no modal
• "Cancel & Replace" is one smooth flow: reason → vendor → done
• Cancelled state is immutable — full audit trail
• Replacement link creates a navigable paper trail
• Same pattern works for "Create Follow-up" on partial completions
Cons
• Slide-out gets taller when cancel form is expanded
• Two buttons (Cancel Only / Cancel & Replace) add decision weight
• Replacement job inherits service + location but admin can't customize fees until it's created

Option B: Dedicated Cancel Modal

Explicit but heavier
Thesis: Cancellation opens a focused modal/dialog (not inline in slide-out). The modal has three clear steps: 1) Enter reason, 2) Choose whether to replace, 3) Confirm. This separates the "destructive action" from the normal slide-out workflow.
Pros
• Cancellation feels appropriately weighty — modal signals "this is serious"
• Clean separation: slide-out = work, modal = destructive action
• More room for cancel reason + replacement details
• Easy to add confirmation step ("Are you sure?")
Cons
• More UI surface — modal on top of slide-out on top of table
• Breaks the "everything in the slide-out" pattern from HMW-49
• Requires closing modal to see the result — more clicks
• For ops doing 50 cancellations, modal fatigue is real

Option C: Two-Step Workflow (Cancel, then manually Create)

Simplest to build
Thesis: Don't combine cancel and replace into one action. Cancel is just cancel (with mandatory reason). If admin wants to replace, they manually add a new job to the shipment using the existing "Add Services" flow. The system links them via a "replaces" field if admin picks the same service type.
Pros
• Simplest implementation — cancel is one action, add job is the existing flow
• No new UI patterns beyond the cancel reason field
• Admin has full control over the replacement job (vendor, fees, everything)
• Reuses existing "Add Job" on the shipment detail
Cons
• Two separate actions — cancel + navigate + add job = friction
• No automatic linkage between cancelled and replacement job
• Easy to forget to create the replacement
• For the 80% case (cancel & replace), this is slow

Recommendation: Option A (Inline Cancel & Replace)

Option A keeps everything in the slide-out — consistent with HMW-49's "one panel, every action" pattern. The cancel form expands inline with the mandatory reason + optional "Create replacement" checkbox. For the 80% case (cancel & replace), it's one flow: reason → pick vendor → done. For simple cancellation, uncheck the box and confirm.

Skip Option B — modal-on-slide-out is heavy and breaks the HMW-49 pattern. For ops doing many cancellations, modal fatigue matters.

Skip Option C — too disconnected. Cancel + manually add replacement = friction and no automatic linkage. The "cancel & replace" case is too common (~80%) to require two separate workflows.

Partial completion: Uses the same "Create Follow-up Job" button on Completed jobs. Admin completes with a remark, adjusts fees (existing capability), then creates a follow-up for the retry. Same linkage pattern (replacedBy/replaces) but labeled "Follows/Follow-up" instead of "Replaces/Replaced by" for clarity.

Key design rules:
• Cancelled jobs are immutable — no vendor swap, no reason edit, no status change
• Cancel reason is mandatory — button stays disabled until textarea has content
• "Cancel Job" button shows as secondary red outline at bottom of slide-out (not in status bar) — available on Pending and In Progress
• Replacement link is navigable — click to open the linked job in the slide-out
• Vendor sees: cancelled status + reason + "Cancelled by Teleport" label (no replacement details)
• Completion remark is a separate concept from cancel reason — amber box, not red
