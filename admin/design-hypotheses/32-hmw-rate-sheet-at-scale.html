HMW-32: Rate Sheet at Scale

How might we keep the vendor rate sheet usable when a vendor serves 15+ routes with 20 L2 fee types — 300+ cells?
The scale problem: HMW-31 introduced the vendor rate sheet — a spreadsheet grid where rows = L2 fee types and columns = routes. At 4 routes, the grid was manageable. But real vendors like HaleSun serve 15+ routes across Shenzhen, Guangzhou, and Hong Kong. That's 20 rows × 15 columns = 300+ cells. Horizontal scrolling becomes brutal — you lose sight of which fee type you're editing.

The real-world pattern: Ops rarely edits ALL routes at once. They typically work on 3-5 routes in a zone (e.g., "update all Shenzhen rates"). The grid needs to support scale without forcing users to scroll through zones they're not working on.

Math: 5 L1 services × 2-6 L2 fees each = 19 rows. 15 routes across 3 zones (Shenzhen: 5, Guangzhou: 5, Hong Kong: 5) = 15 columns. At ~140px per column, that's 2,100px of horizontal content — well beyond any viewport.

Option A: Zone-Grouped Columns + Frozen First Column

Recommended
Thesis: Group route columns by zone and make zone groups collapsible. First column (L2 fee names) is sticky — always visible. Expand only the zone you're working on. This turns 15 columns into 3-5 visible columns at a time, keeping the spreadsheet power without the scroll pain.

How it works:

  • Route columns are grouped under zone headers: Shenzhen (5), Guangzhou (5), Hong Kong (5)
  • Click zone header to expand/collapse — only shows routes in that zone
  • Collapsed zone shows a summary: "5 routes · 42 rates" so you know what's inside
  • First column (L2 fee type) is sticky/frozen — always visible while scrolling
  • Typical workflow: expand Shenzhen, edit 5 routes, collapse, expand Guangzhou, repeat
  • Multiple zones can be open simultaneously if needed for cross-zone comparison
Pros
• Keeps the spreadsheet grid — cross-route comparison within a zone
• Reduces visible columns from 15 to 5-8 at a time
• Frozen first column means you always know what fee you're editing
• Zone grouping matches mental model: "I'm updating Shenzhen rates"
• Collapsed zones act as bookmarks — see where rates exist at a glance
Cons
• Cross-zone comparison requires opening 2 zones (wider view)
• Zone assignment must be maintained in master data
• Slightly more complex UI than flat grid

Option B: Single Route View with Route Picker

Simple
Thesis: Instead of showing all routes at once, show a dropdown to pick ONE route at a time. The rate sheet becomes a simple vertical list: L2 fee type on the left, amount on the right. A "Copy from route" button lets you duplicate rates from an existing route.

How it works:

  • Route dropdown at the top: select one route to view/edit
  • Grid becomes a simple 2-column layout: L2 fee name | rate amount
  • "Copy from route" duplicates all rates from another route as a starting point
  • No horizontal scrolling at all — ever
Pros
• Zero horizontal scrolling — always fits on screen
• Simple, obvious UI — no cognitive overhead
• Works on narrow screens and smaller monitors
• Easy to build and maintain
Cons
• Loses cross-route comparison entirely — can't see "is SZ cheaper than GZ?"
• Must switch routes and remember values for comparison
• Bulk rate entry for a new vendor = N separate sessions (one per route)
• "Copy from route" is a workaround for lacking side-by-side view

Option C: Tabbed Zones

Alternative
Thesis: Each zone (Shenzhen, Guangzhou, Hong Kong) gets its own tab. Within each tab, the grid only shows routes in that zone (3-5 columns). This caps horizontal content at ~700px — always fits without scrolling.

How it works:

  • Tab bar: Shenzhen (5) | Guangzhou (5) | Hong Kong (5)
  • Each tab shows the full 19-row rate grid but only routes in that zone
  • Tab switch is instant — no page reload
  • Can still do cross-route comparison within a zone
Pros
• Predictable column count (3-5 per tab) — never scrolls
• Full grid power within each zone
• Simple mental model: "I'm on the Shenzhen tab"
• Tab badge shows route count — easy overview
Cons
• Cross-zone comparison requires tab switching
• Cannot see "all of HaleSun" in one view
• Adding a new zone = adding a new tab (minor friction)
• Tabs + vendor pills = two levels of selection at the top

Verdict: Option A — Zone-Grouped Columns + Frozen First Column

Zone grouping with collapsible columns is the right pattern because it solves the core problem (horizontal scroll at 15+ routes) without sacrificing the spreadsheet power that makes rate sheets useful.

  • Why not Option B (Single Route)? It kills cross-route comparison, which is a core rate management activity. "Is our Shenzhen rate competitive with Guangzhou?" requires switching and remembering numbers. That's a regression from Excel, not an improvement.
  • Why not Option C (Tabbed Zones)? It's close to Option A but strictly worse. Tabs enforce rigid zone boundaries — you can never see Shenzhen and Guangzhou routes side by side. Option A lets you expand multiple zones when needed.
  • Why Option A wins: Collapsed zones reduce 15 columns to 5-8 visible at a time. Frozen first column keeps fee context. Multiple zones can expand for cross-zone comparison when needed. This is how power users already think: "Let me focus on Shenzhen right now."
Implementation notes:
1. Zone assignment comes from location master data (each location has a zone field)
2. Collapsed zone header shows summary: "5 routes · 42 rates" for at-a-glance coverage
3. First column uses CSS position: sticky; left: 0 — no JS needed
4. Zone expand/collapse state persists in localStorage per vendor
5. Default: first zone expanded, others collapsed
