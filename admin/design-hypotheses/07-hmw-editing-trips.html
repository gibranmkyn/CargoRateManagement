H07: Editing Trips

"How might we let ops fix mistakes without losing the work already done on a trip?"
Scenario

The ops planner created a trip with the wrong customer — it should be AliExpress, not Shopee. Or they need to add a new FM job because the route changed mid-day. The trip already has 2 jobs: one "In Transit" with Jorge, one "Completed" by Hoboken. Deleting the trip and recreating it would lose all status history, activity logs, uploaded proofs, and MAWB tracking. They need to edit the trip in-place while preserving everything already done.

⚠
Business rule: Once a sub-trip (job) is ongoing or completed, it cannot be reassigned — it must be cancelled and a new one created. Pending jobs can be freely edited or deleted. Trip-level fields (customer, MAWB, bags, weight) are always editable until the trip is validated for billing.

Option A: Inline Edit Mode

Recommended
How it works: An "Edit" button on the expanded trip view switches the trip into edit mode. A blue banner appears: "Editing trip — Save / Cancel". Editable fields get dashed blue borders. The distinction between editable and locked is visually clear: pending jobs have blue dashed borders (editable + deletable), in-progress/completed jobs have gray solid borders with a lock icon (read-only). New jobs can be added via a dashed teal "Add Job" card.
Pros
No page navigation — everything stays in context
Explicit edit mode prevents accidental changes
Clear visual distinction between editable and locked elements
Save/Cancel gives a clear "commit" moment for all changes
Cons
Complex state management (edit mode vs view mode)
Edit mode can feel visually noisy with all the dashed borders
Planner must remember to save — unsaved changes could be lost

Option B: Edit Page (Separate Route)

Simple — familiar form pattern
How it works: The "Edit" button navigates to a dedicated edit page (/trips/TRIP20260325-003/edit). The page looks like the Create Trip form but pre-filled with existing data. Locked jobs appear as read-only cards. New jobs can be added at the bottom. A "Save" button at the top saves all changes and returns to the trip detail view.
Pros
Simpler implementation — standard form page pattern
Familiar "edit page" UX — most web apps use this
Full page space for all fields — no cramped inline editing
Clear URL state: /trips/:id/edit vs /trips/:id
Cons
Loses trip list context — planner navigates away from the list
Full page navigation for a small edit (e.g., just fixing the customer name)
Back button confusion — does "back" go to trip detail or trip list?
Feels heavy for minor corrections

Option C: Field-Level Inline Editing

Direct manipulation — fastest for single fixes
How it works: No "edit mode" button. Individual fields are directly clickable to edit. Click the customer name and a dropdown appears inline. Click the MAWB and it becomes a text input. Each field saves immediately on blur or Enter (optimistic update). Locked fields show a lock icon and a tooltip on hover explaining why they can't be edited.
Pros
Fastest for single-field fixes — click, type, done
No mode switching — always "ready to edit"
Feels modern and responsive — Notion/Airtable-style
No need to click Edit, find the field, change it, then Save
Cons
Accidental edits — a stray click triggers edit mode on a field
No clear "save" moment — changes save on blur, which can feel uncertain
Hard to batch multiple changes — each field is independent
Confusing distinction between "this field is clickable" vs "this is read-only"
No undo/cancel for a group of changes

Recommendation: Option A — Inline Edit Mode

An explicit Edit → Save/Cancel flow is the safest and clearest approach for a system where edits have real operational consequences.

  • Prevents accidental changes. Unlike Option C (field-level), the planner must deliberately click "Edit" to enter edit mode. No stray clicks will change the customer name or MAWB.
  • Clear locked vs. editable distinction. Dashed blue borders (editable) vs. gray with lock icon (locked) makes it impossible to confuse which jobs can be modified. This matches the business rule: "once a sub-trip is ongoing, it cannot be reassigned."
  • Stays in context. Unlike Option B (separate page), the planner never leaves the trip view. They can see the trip list behind them, see the full trip detail, and make their edit — all in one place.
  • Batched changes with a clear commit moment. The planner can fix the customer, change the MAWB, and add a new job all in one edit session, then click "Save" once. This is better than field-level saves (Option C) where each change is independent.
  • Unsaved changes warning. If the planner clicks away without saving, a confirmation dialog prevents lost work.
