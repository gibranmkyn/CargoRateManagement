HMW-56: Facility → District Linking

How might we let ops planners link a facility to a specific GB/T 2260 district (3,056 districts across 342 cities, 31 provinces) during facility creation, so the geographic hierarchy is captured for export and reporting?
Current gap: Location type has city, district, districtCode fields but the add/edit form only captures name, code, zone, type. Regions tab has full hierarchy (chinaRegions.ts) but nothing links it into facility creation. The form is inline in a table row — dense ops UI, not a modal.

Verdict

Recommend: A) District Search Dropdown

Why A over B: Ops planners already know where their facilities are — they don't need to browse provinces. A single search field fits the inline table row without width pressure. Three cascading selects (B) demand ~300px and 3 clicks for something these users can type in 2 keystrokes. The inline-row form is already tight at 6 columns; adding 3 selects makes it unworkable.

Why A over C: C's browse fallback adds real implementation cost for a scenario that barely happens. If a user genuinely doesn't know the district name, the Regions tab is one click away. Shipping two interaction modes in one dropdown for a prototype is over-engineering.

Implementation notes for Option A:

  • Search matches Chinese name (宝安区) AND English name (Bao'an District) from the Location seed data's district field — no Chinese reading ability required
  • Results grouped by Province · City for disambiguation (e.g., two 朝阳区 exist: one in Beijing, one in Changchun)
  • Selection auto-fills: district = English name, districtCode = GB/T code, city = parent city name
  • City column in table is read-only / auto-resolved — not a separate input
  • Existing seed data already has correct districtCode values — renders immediately in the table
  • Same search dropdown should also be added to the LocationDropdown inline "Add new location" form
