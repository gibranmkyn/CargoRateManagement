HMW-55

How might we let ops planners filter trips and jobs by a specific date range?

Client feedback: "Add date range filter." Currently: preset period chips (Today / This week / This month / Last month / All time) only appear on All/Completed views. Active view has no date filter.

Current behavior: Active trips show all (no date filter). All/Completed views have preset period chips filtering by createdAt. Client wants explicit date range picking — likely filtering by pickup date, which is operationally more relevant than creation date.
Option A

Date Range Inputs (replace presets)

Replace preset period chips with two date inputs (From / To) in the filter bar. Always visible regardless of status tab. Simple, precise, no ambiguity about what "This week" means.

Option B

Presets + Custom Date Range

Keep existing preset chips (Today / This week / This month / Last month / All time). Add a "Custom" chip that reveals From/To date inputs. Best of both worlds — quick access for common ranges, precise control when needed.

Pros

  • Quick presets survive — "This week" is still one click
  • Custom range available when needed — no information loss
  • Familiar pattern — same filter bar, progressive disclosure

  • Filter bar gets very wide when Custom is active — may wrap on smaller screens
  • 7 period options + Custom is a lot of pills — visual clutter
  • Two mental models (preset vs. custom) — ops has to decide which to use
Option C

Date Range Button + Calendar Popover

Single "Date Range" button in the filter bar. Click opens a calendar popover with range selection + preset shortcuts. Cleanest filter bar, most capable date selection. Familiar from analytics tools (GA, Mixpanel).

How it works:
  • 1. Click date button in filter bar
  • 2. Popover opens with preset shortcuts at top
  • 3. Click preset — immediately applies and closes
  • 4. Or click two dates on calendar for custom range
  • 5. Click "Apply" to confirm custom range
  • 6. Button label updates to show selected range
Date field: Filter by pickup date (operationally relevant), not creation date. Shows trips where any job's pickup date falls within the selected range.

Pros

  • Cleanest filter bar — single button replaces 6 preset chips
  • Presets still available inside popover — no convenience loss
  • Calendar visual makes range selection intuitive (click start, click end)

Cons

  • One extra click to see/change date range (open popover first)
  • Calendar popover is a new component — more implementation effort
  • Popover may feel heavyweight for an ops tool that values speed

Recommendation

Option A — Date Range Inputs is the strongest choice for this ops tool. Two always-visible date inputs are the simplest, most direct solution. Ops planners think in dates ("show me April 1-14"), not in relative periods ("this month"). The inputs are always there — no click to reveal, no popover to navigate.

Option B's preset + custom hybrid adds complexity for marginal benefit — if the user is already typing dates, the presets become redundant. Option C's calendar popover is elegant for analytics tools but adds an extra click that feels heavy in a dense ops workflow where date filtering happens 20+ times a day.

Key change: Filter by pickup date (not creation date). Ops schedules around pickups. The date range should appear on all tabs (Active/Completed/Verified/All) — the current limitation of "date filter only on All/Completed" should be removed. Also update the filter tab split to Active | Completed | Verified | All to match HMW-54's trip-level status derivation.
