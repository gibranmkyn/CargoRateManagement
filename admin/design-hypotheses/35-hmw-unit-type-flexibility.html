HMW-35: Unit Type Flexibility

"Should rate unit types (/trip, /bag, /kg, /cbm) be fixed per Cost ID globally, or configurable per vendor-route?"
The question: FM001 (Warehouse Transfer) is currently /trip (flat rate). But could a different vendor charge FM001 per-kg instead? This decision has a cascading impact on the rate sheet UI, the Services master data, and billing calculations.

If fixed per Cost ID: The rate sheet row label shows "/trip" and all cells in that row are the same unit. Simple, consistent. Every vendor charges FM001 the same way (flat rate), just different amounts.

If flexible: Each cell in the rate sheet grid needs its own unit dropdown. FM001 could be /trip for HaleSun but /kg for Gonda. That's 300+ unit dropdowns in a full rate sheet. More flexible, but noisy and inconsistent.

Real-world signal: Across 5 vendors and 20 Cost IDs, 95%+ of fee types use the same unit for all vendors. Registration fees are always /trip. Handling fees are always /kg. Retrieval is always /bag. The unit is inherent to the fee type, not the vendor.

Option A: Fixed Per Cost ID in Master Data

Recommended
Thesis: Each L2 Cost ID has a default unit type defined once in master data (Services tab). FM001 = /trip, CR002 = /bag, OH003 = /kg. This applies everywhere. The rate sheet row label includes the unit, and all cells in that row are the same unit type. If a vendor genuinely charges a different unit for the same service, create a separate L2 Cost ID.

How it works:

  • Services master data defines L1/L2 hierarchy with unit type per Cost ID
  • Rate sheet rows inherit the unit from master data and display it on the label
  • All cells in a row are the same unit -- just enter the amount
  • Rate sheet inputs are amount-only, no unit dropdown needed
  • Edge case: if a vendor charges per-kg for what's normally /trip, create a new Cost ID (e.g., FM001A "Warehouse Transfer (weight-based)" /kg)
Pros
- Simplest rate sheet UI: amount-only cells, no dropdowns
- Consistent: same fee = same unit everywhere
- Row labels are self-documenting ("FM001 Warehouse Transfer /trip")
- No ambiguity in billing calculations
- Matches 95%+ of real-world rate cards
Cons
- 5% edge case requires creating a new Cost ID
- Slightly more master data management
- Can't accommodate vendor-specific pricing models without a new L2 entry

Option B: Flexible Per Cell

Not Recommended
Thesis: Each rate cell has its own unit type dropdown. FM001 could be /trip for HaleSun on route A but /kg for Gonda on route B. Maximum flexibility, but the rate sheet becomes cluttered with 300+ dropdowns, and it's unclear at a glance what unit a row uses.

How it works:

  • Each cell = amount input + unit dropdown side by side
  • No unit shown on the row label (because it varies)
  • Vendor A might charge FM001 /trip, Vendor B might charge FM001 /kg
  • Billing must check each cell's unit independently
Pros
- Maximum flexibility for edge cases
- No need to create new Cost IDs for different units
- Vendors with non-standard pricing can be accommodated
Cons
- 300+ unit dropdowns clutter the rate sheet
- Inconsistent: same fee, different units across vendors
- Higher error risk (wrong unit selected)
- Billing calculations become per-cell, not per-row
- No visual consistency -- can't scan a row and know the unit

Option C: Default With Override

Compromise
Thesis: Unit type defaults from master data (like Option A) but can be overridden per vendor-route cell. Overridden cells show an amber dot indicator meaning "non-standard unit." This is a compromise between consistency and flexibility.

How it works:

  • Row label shows the default unit from master data (e.g., FM001 /trip)
  • Cells default to the master data unit -- amount-only input
  • Right-click or click a small icon to override the unit for that specific cell
  • Overridden cells show an amber dot: "This cell uses a non-standard unit"
  • Hover the amber dot to see: "Default: /trip. Override: /kg"
Pros
- 95% of cells stay clean (amount only, inherited unit)
- 5% edge cases handled without new Cost IDs
- Visual indicator makes overrides scannable
Cons
- Adds a hidden interaction (right-click / icon to override)
- Override mechanism is non-obvious -- training required
- Mixed units in same row = confusing billing calculations
- Amber dots add visual noise to an already dense grid
- "Is this row /trip or /kg?" becomes ambiguous

Verdict: Option A -- Fixed Per Cost ID in Master Data

Recommended: Option A. Unit types should be fixed per Cost ID, defined once in the Service Catalog (Master Data > Services).

  • Simplest rate sheet: Amount-only cells. No unit dropdowns. No ambiguity. Tab through cells like a spreadsheet.
  • Self-documenting rows: "FM001 Warehouse Transfer /trip" tells you everything. You know every cell in that row is a flat rate.
  • Consistent billing: Rate x qty calculation is the same for every cell in a row. No per-cell unit checking.
  • 95/5 rule: 95%+ of real rate cards use the same unit per fee type across all vendors. The 5% edge case (vendor charges per-kg for a normally /trip fee) is handled by creating a new Cost ID -- which is actually the correct business modeling. A per-kg warehouse transfer is a fundamentally different service.
  • Option B (per-cell) rejected: 300+ dropdowns in a full grid is unacceptable noise. The flexibility isn't worth the cognitive overhead.
  • Option C (default + override) rejected: Adds hidden interactions and amber dots for a 5% case. The complexity isn't justified.
Implementation note: Services master data table needs a "Unit Type" column with values: /trip, /kg, /bag, /cbm. Rate sheet rows inherit the unit and display it on the row label. Rate cells are amount-only inputs. If a new unit type is needed, it's added to the enum in master data config.
