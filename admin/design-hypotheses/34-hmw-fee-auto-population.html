HMW-34: Fee Auto-Population from Rate Cards

When a job is created for FM Trucking with HaleSun on route TikTok WH → SZ Bay, which L2 fees get auto-created as fee line items?
The real scenario: The admin has set up HaleSun's FM Trucking rate card with 3 L2 fees for the TikTok WH → SZ Bay route: FM001 (Warehouse Transfer RM 450/trip), FM002 (Pickup Fee RM 30/trip), FM003 (Cross-Border Handling RM 45/trip). When an ops planner creates a new job on a delivery order and selects HaleSun + FM Trucking + TikTok WH → SZ Bay, what fee line items appear on the job? All 3? Just the primary one? Only the ones marked "default"?

Why it matters: If fees don't auto-populate, the rate card is pointless -- ops has to manually look up and enter each fee, which is exactly what they did in Excel. If ALL fees auto-populate, the job cost is instantly accurate. But if a fee shouldn't apply to a specific shipment, do we want it there by default?

Option A: All L2 Fees With Rates Auto-Populate

Recommended
Thesis: When ops selects a vendor + service + route, the system looks up ALL L2 fee rates configured for that combination in the rate sheet. Every L2 fee that has a rate value creates a FeeLineItem on the job. If FM001=RM 450, FM002=RM 30, FM003=RM 45 for this route, all 3 become fee line items immediately. If FM003 has no rate for a different route, it simply isn't created for jobs on that route.

How it works:

  1. Vendor + Service + Route selected → system queries rate sheet for all L2 fees matching that combination
  2. For each L2 fee with a non-empty rate → creates a FeeLineItem with: name, costId, rate, unit, qty (defaulted), amount (calculated)
  3. Fee line items appear instantly on the create form, showing the full cost breakdown before the job is saved
  4. Ops can remove any fee (click x) or add manual surcharges (+ Add fee) -- but the default is "everything configured"
Pros
• Zero manual fee entry for 95% of jobs
• Job cost is accurate from the moment of creation
• Rate card is the single source of truth
• If a fee shouldn't apply, remove the rate from the sheet -- not per-job
Cons
• Ops might not realize extra fees were added
• Removing a one-time inapplicable fee requires a click
• If rate sheet has errors, wrong fees propagate to many jobs

Option B: Only Base Rate Auto-Populates, Others Manual

Conservative
Thesis: Only the first/primary L2 fee (e.g., FM001 Warehouse Transfer) auto-creates a fee line item. Other L2 fees (FM002 Pickup, FM003 Cross-Border Handling) are available via an "Add fee" dropdown but must be manually selected. This gives ops explicit control over which fees apply to each job.

How it works:

  1. Vendor + Service + Route selected → system creates only 1 FeeLineItem from the "primary" L2 fee (first in the rate card)
  2. Ops clicks "+ Add fee" → dropdown shows remaining L2 fees with their rates pre-filled. Select to add.
  3. Benefit: Ops explicitly approves each fee. No surprise charges.
  4. Cost: 2 extra clicks per job for fees that apply 99% of the time.
Pros
• Ops has explicit control over each fee
• No surprise charges on jobs
• Simpler initial experience -- only 1 fee visible at first
Cons
• Defeats the purpose of a comprehensive rate card
• 2 extra clicks per job for fees that almost always apply
• Ops might forget to add mandatory fees (e.g., cross-border handling)
• "Which fee is the primary one?" -- ambiguous for services with 5+ L2 fees

Option C: Configurable Per Vendor-Service -- "Rate Card Template"

Maximum Config
Thesis: In the rate sheet, admin marks which L2 fees are "default" (auto-populate) and which are "optional" (available but not auto-created) for each vendor-service combination. When a job is created, only default-marked fees become fee line items. Optional fees are available via "+ Add fee" with rates pre-filled.

How it works:

  1. In the rate sheet: each L2 fee row has a toggle: "Default" (auto-create) or "Optional" (available on demand)
  2. On job creation: only L2 fees marked "Default" become FeeLineItems
  3. "+ Add fee" shows optional L2 fees with rates pre-filled, plus manual entry
  4. Example: Admin marks FM001 and FM003 as default, FM002 as optional. Job gets 2 auto-fees, FM002 available on demand.
Pros
• Maximum flexibility -- admins decide per vendor what's standard
• Different vendors can have different default fees
• Seasonal or route-specific fees can be toggled
Cons
• More config to manage in the rate sheet
• "Default vs Optional" adds a column to every rate row
• Admin overhead: must review defaults when adding new L2 fees
• Complexity for a problem that might not exist (are there fees that only sometimes apply?)

Verdict: Option A -- All Configured L2 Fees Auto-Populate

Recommended because: The entire point of rate cards is to pre-define the cost structure for a vendor-service-route combination. If the admin put FM001, FM002, and FM003 in HaleSun's FM rate card for the TikTok WH → SZ Bay route, those fees should all appear on every job for that combination. Period.

  • Rule: Rate in the sheet = fee on the job. No rate = no fee. Simple.
  • Control point is the rate sheet, not the job. If FM002 shouldn't apply to a route, remove its rate from that column. Don't make ops decide per-job.
  • Option B forces ops to manually add fees that apply 99% of the time. That's Excel behavior -- exactly what we're replacing.
  • Option C adds a "default/optional" toggle that solves a theoretical problem. In practice, if a fee has a rate for a route, it applies. If it doesn't apply, delete the rate.
  • Edge case: If a fee genuinely doesn't apply to a specific shipment, ops can remove it (click x). This is rare and the 1-click removal is less friction than 2-click manual addition on every job.
  • Job creation flow: Select vendor + service + route → all matching L2 fees appear instantly with rates and amounts → total cost visible before saving. Zero manual fee entry for standard jobs.
