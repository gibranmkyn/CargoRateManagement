HMW-28: Active vs Historical Delivery Orders

How might we help ops planners focus on the orders that need attention right now, while keeping historical orders accessible when needed for reference or billing?
The daily reality: An ops planner managing 30 orders/day cares about ~15-20 active orders (Pending + In Progress). Completed orders pile up — after a week there are 100+. Right now everything is in one flat list. The planner scrolls past completed TikTok orders from last week to find the Shopee order that needs attention now. The mental model is: "Show me what needs work. Let me check old stuff when I need to."

Option A: Smart Default Filter — "Needs Attention" as Default

Recommended
Thesis: The filter bar gets a new row of status filter chips: Active (default, auto-selected on page load) | All | Completed. "Active" shows orders that have at least one job NOT completed. The existing MAWB/Customer/Vendor filters work on top. No tabs, no separate pages — just a pre-applied filter that matches the daily workflow.

Why "Active" not "Needs Attention": "Needs Attention" implies urgency/problems. Most active orders are normal — they just aren't done yet. "Active" is neutral and accurate.

✓ Pros
• Zero learning curve — it's just a filter, already familiar
• Default matches 90% use case (active orders)
• One click to see "All" or just "Completed"
• Stats bar updates to reflect filtered view
• URL can encode filter state (?status=active)
✗ Cons
• New users might wonder "where are my completed orders?"
• Filter state resets on page reload (unless persisted)

Option B: Tabs — Active | History

Tab Pattern
Thesis: Two tabs at the top of the page: Active (default) and History. Active shows orders with at least one non-completed job. History shows only fully completed orders. Separate views, separate mental modes. Similar to how email has Inbox vs Archive.
✓ Pros
• Clear mental separation: "work" vs "archive"
• History tab can have different columns (e.g., completion date, total billed)
• Familiar pattern (email inbox/archive)
• Stats bar shows different metrics per tab
✗ Cons
• Tabs add a navigation layer — slightly more complex
• "Where did my order go?" when it moves from Active to History
• 2 views to maintain (different table layouts)

Option C: Grouped — Active First, Completed Collapsed Below

Single View
Thesis: One continuous list, but orders are grouped: Active orders at the top (sorted by pickup date), then a collapsible "Completed" section at the bottom. Similar to how task managers show active tasks above and completed tasks in a muted, collapsed section. No context switching — scroll down to find old stuff.
✓ Pros
• Everything on one page — no "where did it go?"
• Completed section is naturally de-emphasized
• Can peek at completed without switching views
• Simple mental model: top = work, bottom = done
✗ Cons
• Completed section grows unbounded over time
• Page gets long as history accumulates
• Grouping logic adds rendering complexity
• Collapsed section might never be opened

Verdict: Option A — Smart Default Filter

Recommended because: It's the smallest change with the biggest impact. The table, columns, and interactions stay exactly the same — we just add status filter chips to the existing filter bar and default to "Active." Ops planners already understand filters. No new patterns to learn, no mental model shift, no "where did my order go?" confusion.

  • "Active" = orders where at least 1 job is NOT Completed (Pending, In Progress, or Rejected)
  • "Completed" = all jobs in the order are Completed
  • "All" = everything (current behavior)
  • Stats bar updates to reflect the filtered count
  • Default on page load: Active. Persists in URL as ?status=active
  • Count badges on each chip: Active (6) | All (10) | Completed (4)
