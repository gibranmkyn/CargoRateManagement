H11: Order Priority Visibility

"How might we help ops instantly see which orders are urgent -- so they triage the right ones first?"
Problem: Some orders have "Priority shipment" in remarks, but it looks the same as every other row. Ops scanning 50 rows can't tell which ones need attention first. A priority TikTok shipment with a tight SLA sits visually identical to a standard Shopee order picking up 3 days from now.

Option A: Priority Tag in Customer Cell

Recommended
How it works: A small colored tag "PRIORITY" appears next to the customer name in the same cell, using a warm amber/orange color (#b45309 text on #fefce8 bg with #fde68a border). Non-priority orders show nothing -- clean and quiet.
Pros
Visible at a glance without a full extra column
Doesn't add visual noise to non-priority rows
Works with existing dense layout -- no column resize needed
Amber color is distinct from status colors (green/red/gray)
Cons
Customer cell gets wider if name is long + tag
Only visible when scanning the customer column
Might be missed if ops scans right-to-left

Option B: Full-Row Tint

Ambient signal
How it works: Priority rows get a subtle warm background (#fffbeb) -- the same way rejected rows get #fefafa. No tag or text needed; the background itself signals urgency. The warm amber tint contrasts with the cool red of rejected rows.
Pros
Entire row stands out -- visible from any column
No extra UI elements or column width changes
Consistent pattern with rejected-row tinting
Very subtle -- doesn't clutter the interface
Cons
Subtle can mean invisible -- ops may not notice the tint
No label explains WHY the row is tinted (new users confused)
Could clash with hover/selection states
Colorblind users may struggle to distinguish amber from red tint

Option C: Priority Column with Icon

Explicit but uses space
How it works: Add a narrow column (28px) after the chevron with a flag icon for priority orders. Non-priority rows show an empty cell. The flag icon uses the same amber color as Option A's tag.
Pros
Unmistakable -- dedicated column is the clearest signal
Sortable -- click column header to sort priority-first
Scannable -- eyes can check the column vertically in one pass
Works for colorblind users (icon + color)
Cons
Uses horizontal space in an already dense table
Empty cells for 90% of rows -- wasted column
Adds visual complexity to the header row
May feel over-engineered for a boolean flag

Recommendation: Option A -- Priority Tag in Customer Cell

The inline tag is the best balance of visibility and economy. It is immediately visible when scanning the customer column (which ops reads first anyway), uses the amber palette that is already reserved for warnings/attention in the design system, and adds zero visual noise to the 90%+ of rows that are non-priority. Option B is too subtle -- ops will miss it. Option C wastes horizontal space for what is ultimately a boolean flag on a minority of rows. The tag approach also composes well with other future row-level signals (e.g., "VIP" tags, "FRAGILE" tags) without needing a new column for each.
