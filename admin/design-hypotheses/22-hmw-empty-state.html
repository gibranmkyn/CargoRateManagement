HMW: Empty State

"How might we welcome a first-time ops planner so the empty app feels like an invitation, not a broken page?"
Current problem: Day 1, a new ops planner opens Teleport OS. No delivery orders exist yet. They see: a stats bar showing all zeros, an empty table with column headers, and nothing else. No guidance on what to do first. The app looks broken -- like data failed to load, not like a fresh workspace.

Who this affects: New Teleport ops team members, new customer onboarding (when the ops planner sets up the first TikTok or Shopee delivery), and demo scenarios where the app needs to look intentional even without data.

Constraints: Ops planners are logistics professionals, not consumer app users. They don't need onboarding tours or tooltips. They need a clear signal: "this is working, here's how to start." The empty state should feel professional and dense -- not like a SaaS marketing page.

Option A: Empty Table with Inline CTA

Recommended
How it works: The page layout stays exactly the same: nav, stats bar, table with column headers. But the table body -- where data rows would be -- shows a centered empty state message: a teal ship icon, "No delivery orders yet," a one-line explanation, and a [+ New Order] button. The stats bar shows "0" for everything (honest, not hidden). When the first order is created, the empty state disappears and data rows replace it. No special logic to dismiss.
Pros
- Minimal -- doesn't change the page structure at all
- The empty state IS the table body, so it disappears naturally when data exists
- Clear CTA: the [+ New Order] button is the obvious next action
- Stats bar with zeros is honest and functional (no fake data, no hiding)
- No "dismiss" logic needed -- data replaces the empty state
Cons
- Doesn't explain what an "order" is or how the 3-step workflow works
- For someone truly unfamiliar with Teleport OS, the CTA is "create an order" but they don't know what that entails
- Slightly plain -- professional, but not inspiring

Option B: Getting Started Guide

Educational
How it works: Instead of the empty table body, show a 3-step "getting started" card that explains the Teleport OS workflow. Step 1: "Create a delivery order" -- assign customer, MAWB, and services. Step 2: "Track job progress" -- vendors execute, you monitor status changes. Step 3: "Validate for billing" -- upload proofs, validate completed orders. A [Get Started] button at the bottom takes the user to the create order form. The card disappears permanently after the first order is created.
Pros
- Educates new users on the full workflow (create → track → validate)
- Feels intentional -- the empty state was designed, not neglected
- Single CTA is clear: "Get Started"
- Disappears after first order -- returning users never see it again
Cons
- Takes more vertical space than a simple message
- The "track" and "validate" steps are things ops will learn by doing, not by reading
- Feels slightly hand-hold-y for logistics professionals
- Requires "dismissal" logic: must track whether user has created first order

Option C: Sample Data with Dismiss

Exploratory
How it works: Pre-load 3 sample delivery orders with realistic data (TikTok, Shopee, Shein). A dismissible banner at the top says "These are sample orders to help you explore Teleport OS. [Dismiss samples]." Each sample row has a "SAMPLE" tag next to the order ID. Ops can click into samples to see how the slide-out panel, status changes, and proof uploads work. Dismissing clears all samples and shows the clean empty state (Option A's inline CTA).
Pros
- User sees a fully working app immediately -- learns by exploring, not reading
- Can practice status changes, proof uploads, etc. without creating real data
- Most realistic introduction to the tool's density and data patterns
- Dismissal is one click -- not persistent
Cons
- Risk of confusion: ops might think sample data is real and try to act on it
- More complex to implement (seed data, dismiss logic, cleanup)
- If ops doesn't dismiss, samples pollute the real data view
- Sample data needs to be maintained as the app evolves

Verdict: Option A -- Empty Table with Inline CTA

The inline CTA in the empty table body is the densest, most professional approach. It doesn't change the page structure, doesn't require dismiss logic, and doesn't patronize ops planners with onboarding tutorials. The stats bar showing all zeros is honest -- ops knows they're looking at a fresh workspace, not a broken one. The [+ New Order] button is the single, clear next action.

Why not Option B? The 3-step guide is educational but too hand-hold-y for logistics pros. They'll learn the workflow by doing, not by reading a card. The extra vertical space doesn't earn its keep. Why not Option C? Sample data is the most engaging but also the riskiest -- ops might confuse samples with real orders, and the implementation cost (seeding, cleanup, dismiss) is high for a state that only lasts until the first order is created. Option A is the utilitarian choice: professional, clear, and self-removing.
