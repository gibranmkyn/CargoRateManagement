HMW: Rate Management

"How might we incorporate vendor rate cards into the delivery order workflow — so ops sees costs while booking, and billing admin can validate invoices against agreed rates?"
Rate structure:
• CR, CS, EC, OH → rate per vendor per location (e.g., Gonda charges RM 150 for EC at Shenzhen Bay)
• FM Trucking → rate per vendor per origin-destination pair (e.g., HaleSun charges RM 450 from TikTok WH → SZX Airport)

Why this matters: Without rate visibility, ops creates orders blind to cost. Billing admin reconciles invoices manually against spreadsheet rate cards. Rate disputes are common — vendor says RM 500, Teleport expected RM 450. A rate management system prevents this.

Proposed Data Model

VendorRate { vendorCode: string // "V-001" (HaleSun) serviceCode: string // "FM", "EC", "CR", "CS", "OH" rateType: "location" | "route" // For location-based (CR, CS, EC, OH): location?: string // "Shenzhen Bay Checkpoint" rate: number // 150.00 // For route-based (FM): origin?: string // "TikTok WH, Shenzhen" destination?: string // "SZX Airport Cargo Terminal" rate: number // 450.00 currency: "MYR" | "CNY" | "USD" effectiveFrom: string // "2026-01-01" effectiveTo?: string // null = still active }

Option A: Inline Rate Display

Recommended
How it works: Rates appear inline wherever vendors and services are shown — in the create form, in the sub-table, and in the slide-out panel. No separate "Rate Management" page for Phase 1. Rates are loaded from a rate card JSON file (same pattern as vendor/customer master data). When ops selects a vendor + service + location, the rate auto-populates. Each order shows estimated total cost.
✓ Pros
• Zero navigation — rates visible in context
• Ops sees cost while creating the order
• Sub-table shows rate per job + order total
• Simple to implement — just lookup + display
✕ Cons
• No rate editing UI (admin edits JSON/master data directly)
• Can't see rate history or compare vendors by rate
• Rate disputes still need manual investigation

Option B: Rate Management Page + Inline Display

Full solution
How it works: Everything in Option A, plus a dedicated "Rates" page in the nav where admin can view, add, edit, and deactivate vendor rates. Two sub-tabs: "By Vendor" (see all rates for HaleSun) and "By Service" (see all FM Trucking rates across vendors).
✓ Pros
• Full rate lifecycle management in the app
• Admin can compare rates across vendors (who's cheapest for FM from Yantian?)
• Rate history — see when rates changed
• Can flag anomalies (vendor charging above agreed rate)
✕ Cons
• More pages to build (Rate list, Rate add/edit form)
• More data model complexity (rate history, effective dates)
• Scope creep risk — could become a full procurement system

Option C: Rate Suggestion Engine

Future vision
How it works: Everything in Option B, plus the system suggests the cheapest vendor for each service based on rate cards. When creating an order, ops sees "HaleSun RM 450 | SevenSeas RM 420 | The Lorry RM 510" and can pick the best option.
✓ Pros
• Ops makes cost-optimal decisions
• Reduces back-and-forth on vendor selection
• Could save 10-20% on transport costs
✕ Cons
• Requires complete rate cards for all vendors (data quality issue)
• Cheapest isn't always best (reliability, availability matter)
• Complex to implement — needs vendor capacity awareness too

Recommendation: Start with Option A, build toward B

Phase 1: Option A (inline rate display). Add rate lookup to the create form and sub-table. Rates stored as JSON constants (same as vendors/customers). Ops sees estimated cost per job + order total. Takes ~1 day to implement.

Phase 2: Option B (rate management page). Build the "Rates" nav section where admin can CRUD vendor rates. Add the rate anomaly detection to the Validation page. This is a natural extension once the data model is in place.

Phase 3: Option C (suggestion engine). Only after rate cards are complete for all vendors. This is the "10x" feature but requires data quality to be high first.

Data model impact: Add `VendorRate` entity to mockData.ts. Add a `rate` field to the Job interface (populated from lookup during order creation). The rate column in the sub-table shows per-job cost. The order header shows total.
