HMW-39: FTL Rate Management

How might we design the FTL trucking rate management so ops admins can view and edit route × truck type rates in the same format vendors quote them — while connecting to the Regions master data and supporting truck type selection during order creation?
User stories addressed: US-020 to US-029

Real data from vendor rate cards:
  • 164 routes across 2 vendors (HaleSun + Gonda)
  • Truck types: 1.5T, 3T, 5T, 8T, 10T, 12T, 40HQ, 45HQ (not all available on all routes)
  • Origin: City + District (深圳·福田街道, 东莞·洪梅镇, 广州·花都区)
  • Destination: Fixed hub per vendor (HaleSun → 深圳机场, Gonda → 香港机场仓库)
  • Currency: RMB for domestic, HKD for cross-border
  • Accessorials: FM001 (Warehouse Transfer), FM002 (Pickup), FM003 (Cross-Border) are add-ons, not the base FTL rate

Design Decisions

  • Rates page split: "🚛 Trucking (FTL)" tab and "📦 Services (L2 Fees)" tab. Trucking shows route × truck type. Services shows L2 Cost ID × location.
  • FTL rate sheet: Rows = origin districts grouped by city. Columns = truck types with capacity specs. Sticky first column. City groups collapsible.
  • Destination: Shown as a header above the grid (per vendor, most vendors have 1-2 fixed hubs). Not a per-row field.
  • Origin districts: From Regions master data (GB/T 2260 codes). "+ Add route from Regions" to add new rows.
  • Truck type in create form: Chip buttons (1.5T / 3T / 5T / 8T / 10T / 40HQ) next to vendor dropdown. Click to select. Selected type highlighted in blue.
  • Origin/destination in create form for FM: District-level pickers (City → District cascade) instead of facility pickers used by other services.
  • Fee model: FTL base rate (from truck type sheet) + FM accessorial fees (FM001/FM002/FM003 from Services tab). Both auto-populate. Base rate is the primary cost; accessorials are add-ons.
  • Rate lookup key: vendorCode + originDistrictCode + destinationHub + truckType → flat rate per trip.
  • Data model change: FTL rates need a truckType field and originDistrictCode / destinationDistrictCode fields (GB/T codes), distinct from the current VendorRate which uses locationId.
