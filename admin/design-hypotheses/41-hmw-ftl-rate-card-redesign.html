HMW-41: FTL Rate Card — Redesigned

How might we design the FTL rate card so it feels like a living document — with identity, versioning, audit trail, quick editing, and bulk CSV import — not just a flat table?
Design audit findings (Flexport lens):
  1. A rate card is a document, not a filter. "HaleSun → Shenzhen Hub" is the card's identity. Don't put the destination in a dropdown.
  2. Allow inline editing for quick fixes. CSV for bulk, inline for tweaks. Don't force a 5-step CSV roundtrip for changing one number.
  3. Preview should summarize, not list 1,300 rows. Group by city: "Shenzhen: 48 updated, 4 new."
  4. Rate cards need metadata. Version, effective date, who uploaded, activity log.
  5. Versioning is essential. When vendor sends Q2 rates, Q1 rates should be preserved. "View previous version" for disputes.

Design Decisions

  • Rate card = document. Each vendor → destination pair is a named rate card with version, metadata, and activity log. Not a filter on a flat table.
  • Landing page shows cards as a grid. FTL cards and Service cards separated. Click a card → drill into the rate sheet.
  • Rate card header is a dark banner with vendor → destination identity, version, stats, and CSV actions. Not a dropdown.
  • Version history. Each CSV upload creates a new version. Previous versions accessible. "View v2" for dispute resolution.
  • Inline editing for quick fixes. Click a cell → edit in place. Unsaved changes bar at bottom. Each save is logged in activity.
  • CSV for bulk. "Upload New Version" creates a versioned bulk update. Preview shows summary by city (not 1,300 individual rows). "Show individual route changes" expandable for detail.
  • Activity log on every card. Every change tracked: CSV uploads (with filename, stats), inline edits (old → new value), card creation. Timestamp + user for every entry.
  • No destination dropdown. Destination is the card's identity. If a vendor adds a second hub, that's a new card.
