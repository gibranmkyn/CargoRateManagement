HMW-31: Rates Module Redesign with L1/L2 Services

How might we redesign the rates module so it's structured around the L1/L2 service hierarchy, supports bulk rate entry, and feels delightful instead of tedious — while properly connecting to master data?
What's wrong today: The rates page is a flat table of rates grouped by vendor. Adding rates is one-by-one via a slide-out form. There's no concept of L2 sub-services — each rate is just "vendor + L1 service + location = amount." But the real business has 20 distinct fee types (Cost IDs) across 5 services, and a rate card should define ALL applicable L2 fees for a vendor-route combination at once.

Option A: Vendor Rate Sheet — Spreadsheet-Style Bulk Entry

Recommended
Thesis: Instead of adding rates one-by-one, the rates page shows a "rate sheet" per vendor — a spreadsheet-like grid where columns are routes/locations and rows are L2 fee types (Cost IDs). Admin fills in the grid, hits save, and all rates are created at once. This mirrors how logistics rate cards actually work in the real world — they're spreadsheets.

How it works:

  • Select a vendor → see their rate sheet
  • Rate sheet rows = L2 fee types (grouped by L1 service)
  • Rate sheet columns = routes/locations where this vendor operates
  • Each cell = rate amount + unit. Blank = not applicable
  • Edit any cell inline. Tab between cells like a spreadsheet
  • "+ Add route" to add a new column
  • Save all changes at once
✓ Pros
• Mirrors real-world rate card format (Excel)
• Bulk entry: fill 20 rates in 2 minutes, not 20 forms
• L2 structure is visible and obvious
• At-a-glance: see all rates for a vendor across all routes
• Copy rates between routes (common operation)
✗ Cons
• More complex to build than a simple table
• Horizontal scrolling if many routes
• Need to handle sparse grids (most vendors don't serve all routes)

Option B: Vendor Profile Cards with L2 Accordion

Card Pattern
Thesis: Each vendor gets a "profile card" showing their L1 services. Click an L1 service to expand and see all L2 fees with rates. Add rates per L2 fee inline. This is less spreadsheet-like but more visually organized.
✓ Pros
• Visually organized — L1/L2 hierarchy is clear
• Less overwhelming than a spreadsheet grid
• Each vendor's capability is visible at a glance
✗ Cons
• Still adding rates one-by-one per L2 fee
• Can't compare rates across routes easily
• More clicking (expand L1 → expand L2 → edit)

Verdict: Option A — Vendor Rate Sheet

Recommended because: Logistics ops already thinks in rate card spreadsheets. The grid format is the native language of rate management — rows are fee types, columns are routes. Bulk editing means configuring a vendor's entire rate card takes minutes, not hours of one-by-one form submissions.

  • L2 fee taxonomy lives in master data (Services tab) — 20 Cost IDs across 5 L1 services
  • Rate sheet per vendor — grid of L2 fees × routes/locations, inline editable
  • Dark column headers with route info — matches the Teleport brand nav aesthetic
  • "+ Route" button to add new route columns
  • Unit type per L2 fee — /trip, /bag, /kg are fixed per Cost ID, not per rate
  • Currency per cell — some routes are CNY, others MYR
  • Empty cells = not applicable — sparse grids are fine, not every vendor charges every fee on every route
  • Save all at once — no individual form submissions

Master data connection: The Services tab in Master Data needs to be updated to show the L1/L2 hierarchy with Cost IDs. This is the source of truth that generates the rate sheet rows. Adding a new L2 fee in master data automatically adds a new row to all vendor rate sheets.
