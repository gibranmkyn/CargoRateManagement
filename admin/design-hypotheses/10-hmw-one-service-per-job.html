HMW: One Service Per Job

"How might we structure jobs so that each one tracks exactly 1 service — making billing, status, and proof tracking unambiguous?"
User's requirement: "Each job should only have 1 service code. If the same vendor handles trucking AND customs, those should be 2 separate jobs — not 1 job with 2 services."

Why this matters for billing: Each service maps to an L1/L2 cost ID. If a job bundles FM+EC, which service is the proof for? Which service is completed vs pending? The billing team needs service-level granularity.

Option A: 1 Job = 1 Service (Proposed)

Model change: `Job.services` (array) → `Job.service` (single ServiceType). If vendor Camila does FM Trucking AND EC Customs, that's 2 jobs: J01 (Camila · FM) and J02 (Camila · EC). Each job tracks its own status, proof, and activity log independently.
✓ Pros
• Billing is unambiguous — 1 job = 1 cost line item
• Status per service: trucking can be "completed" while customs is still "in progress"
• Proof per service: delivery receipt for trucking, clearance doc for customs
• Simpler data model — no need for ServiceMultiSelect component
• Sub-table view is cleaner — each row IS a billable line item
✕ Cons
• More jobs per order (was 2-3, now 3-5 for complex orders)
• Same vendor appears multiple times in the table
• Create form changes — add 1 job per service instead of grouping
Shopee PTRIP12302100002 16 bags · 890 kg
Job Vendor Service Origin → Destination Status Proofs
J01 Hoboken CR Cargo Retrieval Yantian Port → Dongguan Completed 1 doc ✓
J02 Hoboken FM Trucking Yantian Port → Dongguan In Progress 0 docs
J03 V3 EC Export Customs Customs clearance (parallel) Pending 0 docs
Notice: Hoboken appears twice (J01 + J02) — same vendor, different services. Each has its own status and proof column. The billing team sees 3 line items, each with a clear cost ID.

Option B: 1 Job = Multiple Services (Current)

Current model: `Job.services` is an array. Hoboken handles CR+FM as one job. Service pills are shown together.
✓ Pros
• Fewer jobs per order (simpler table)
• Groups vendor assignments naturally
• Create form allows multi-select services per vendor
✕ Cons
• Billing is ambiguous — which service in the bundle is the proof for?
• Status is per-job, not per-service — trucking and customs share one status
• If trucking is done but customs isn't, what status does the job show?
• The original CR requirement was 1 service per child trip

Impact on Create Form

Option A: Simpler creation flow
Step 2 stays the same: check off required services (FM, EC, CS...).
Step 3 changes: instead of grouping services into jobs, each service IS a job.
For each selected service → assign a vendor + origin + destination.

The form becomes:
J01 FM Trucking → Hoboken Yantian → Dongguan
J02 CR Cargo Retrieval → Hoboken Yantian → Dongguan
J03 EC Export Customs → V3 Customs clearance
Each row is: service badge → vendor dropdown → route. No multi-select needed. Cleaner, faster.

Recommendation: Option A (1 Job = 1 Service)

This aligns with the user's explicit requirement and the original CR spec. It makes billing unambiguous, allows per-service status tracking and proof uploads, and simplifies the create form. The trade-off (more rows in the sub-table) is actually a benefit — each row IS a billable line item. The sub-table becomes the billing worksheet.

Data model change: `Job.services: ServiceType[]` → `Job.service: ServiceType` (single). ServiceMultiSelect component replaced with a simple service dropdown per job.
