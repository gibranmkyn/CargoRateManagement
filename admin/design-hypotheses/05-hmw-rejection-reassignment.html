H05: Rejection & Reassignment

"How might we surface rejected jobs so ops can reassign the vendor within seconds — before cargo sits idle?"
Scenario

Camila's truck broke down. She rejects the FM+CS job for TRIP20260325-003. There are 36 bags of AliExpress priority cargo sitting at Qianhai FTZ with no vendor assigned. Every minute the cargo sits idle costs money, and the customer's SLA timer is ticking. The ops planner needs to see this immediately, understand what happened, and reassign the job to another vendor — ideally within 30 seconds.

Option A: Alert Banner + Red Row Highlight

Recommended
How it works: When a vendor rejects a job, a red alert banner appears at the top of the trip list: "1 rejected job needs reassignment" with a "View" link. The rejected trip row gets a red left border and a "NEEDS ACTION" tag. Clicking the row opens the standard slide-out panel where the planner can reassign the vendor from a dropdown.
Pros
Non-intrusive — fits existing table layout
Red banner is visible without being aggressive
Reuses the existing slide-out panel (no new patterns)
Planner can finish current task before responding
Cons
Banner can be dismissed or ignored
No urgency beyond color — no sound or forced interaction
If planner is on a different page, they won't see it until they return to trip list

Option B: Blocking Modal

Aggressive — forces immediate action
How it works: The moment a vendor rejects a job, a modal appears for the ops planner. The modal shows the rejection reason, route, services, and cargo details. It includes a vendor dropdown to reassign immediately. The modal cannot be dismissed without taking action — the planner must either reassign the job or snooze for 15 minutes.
Pros
Cannot be missed — guaranteed immediate awareness
Forces resolution — no chance of forgotten rejections
Fastest possible time-to-reassignment
All context is in the modal — no navigation needed
Cons
Interrupts whatever the planner is doing right now
Feels aggressive — bad UX if planner is mid-task
If multiple rejections happen at once, stacked modals are terrible
Creates anxiety and pressure

Option C: Dedicated "Needs Attention" Queue

Separate view — clean but requires navigation
How it works: A new tab on the trip list: "Needs Attention (1)" alongside "All Trips". This tab shows only trips with rejected or problematic jobs. Each item in the queue shows full context — rejection reason, route, cargo, and an inline vendor reassignment dropdown — so the planner can resolve issues without navigating to individual trips.
Pros
Clean separation — main trip list stays uncluttered
Full context per item — no need to click into anything
Works as a daily "to-do" list for the planner
Scales well if there are multiple issues at once
Cons
Requires switching to the tab — planner might not check it
Rejection could go unnoticed until the planner clicks the tab
Another view to maintain and learn
No push notification — relies on the count badge being noticed

Recommendation: Option A — Alert Banner + Red Row Highlight

The banner + red row approach is the best balance between visibility and non-intrusiveness.

  • Non-intrusive but unmissable — the red banner sits at the top of the trip list, visible the moment the planner looks at the screen, but doesn't block their current workflow.
  • No new patterns to learn — clicking the row opens the existing slide-out panel, which already has the vendor dropdown and job details. The reassignment flow is the same as the normal assignment flow.
  • Graceful degradation — if multiple rejections happen, the banner updates the count ("3 rejected jobs") and multiple rows light up red. No stacking modals, no cluttered queue.
  • Option B (modal) is too aggressive — it interrupts flow and creates anxiety. If the planner is creating a trip for a different customer, a blocking modal is hostile.
  • Option C (queue) is too passive — it relies on the planner checking the tab. The red count badge might not be enough urgency for time-sensitive cargo.
  • Future enhancement: Add browser notification / sound alert for high-priority rejections so planners on a different page get notified.
