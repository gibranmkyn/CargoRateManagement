HMW-53

How might we let ops planners edit trip details and job assignments after creation without disrupting their scan-and-act workflow?

Current gap: After creation, trip-level fields (customer, MAWB, dates, route, cargo) and job-level fields (vendor, service, route) are immutable. The only recourse is cancel & recreate. Ops planners need to fix mistakes and update changing plans without losing audit trail.

Option A

Inline Row Edit — "Edit where you see"

Click the pencil icon on any trip row → an edit bar expands below the row with all trip fields as inputs. Save or cancel inline. Job fields become editable in the slide-out panel via a similar toggle. Fastest for single-field corrections.

Pros

  • Fastest for single-field corrections — no navigation, no page change
  • Spreadsheet-native — ops planners already think in rows and cells
  • Maintains scanning context — other rows stay visible while editing

Cons

  • Cramped for trips with many fields — 10 inputs in a narrow bar
  • Can't add or remove jobs from the inline edit — only trip-level fields
  • Two different edit surfaces (row for trip, slide-out for job) — split mental model
Option B

Edit Page — "Reuse the creation form"

Click "Edit" on a trip → navigate to /trips/:id/edit which reuses the same 2-step form layout as Create Trip, pre-filled with existing data. Edit all trip fields + add/remove/modify jobs in one view. Maximum power, full context switch.

Pros

  • Maximum power — can edit everything: trip fields + add/remove/modify jobs in one view

Option C

Section Edit Toggles — "Pencil icon per section"

Each section (trip header, job route, job vendor) gets its own pencil icon. Click it → just that section becomes editable. Save per section. Most granular, stays within existing UI surfaces (expanded row for trip, slide-out for job). No page navigation.

Pros

  • Most granular — edit just what you need (MAWB only, or route only) without touching other fields
  • No page navigation — stays within trips table and existing slide-out surface
  • Clear editability boundaries — each section independently toggles, with section-level save/cancel

Cons

  • Many separate save buttons — if ops needs to change 3 sections, that's 3 save clicks
  • More complex UI state — tracking which sections are in edit mode, preventing conflicting saves
  • Cannot add or remove jobs from this surface — still need a separate action for structural changes
RECOMMENDATION

Option A (Inline Row Edit) for trips + Option C (Section Toggles) for jobs in the slide-out

Rationale: The dominant edit case is quick corrections — fixing a MAWB, updating a pickup date, correcting bag count. These are 1-2 field changes, done 5-10x/day. Option A's inline row edit is the fastest path for trip-level changes: click pencil, fix the field, save. No navigation, no context loss. It matches the "smart spreadsheet" aesthetic — Excel users expect to click a cell and type.

For job-level edits (vendor, route, dates), the slide-out already owns job detail — adding section-level edit toggles (Option C's pattern) is a natural extension. Each section gets a pencil icon; click it, edit that section, save. This avoids the split mental model problem of Option A's pure approach while keeping the granularity.

Option B (Edit Page) is overkill for the 90% case and breaks scanning flow. However, it's the right answer for adding/removing jobs after creation — which is a rarer, more deliberate action. We could add a small "Manage Jobs" link in the expanded sub-table that navigates to the edit form only for structural changes (add/remove jobs), while trip field edits and job field edits stay inline.

Editability rules: Trip fields editable anytime (customer, MAWB, dates, route, cargo, remarks). Job fields editable only when job is Pending or In Progress — Completed/Verified/Cancelled jobs are locked. Changing a job's vendor or route recalculates fees from the schedule. All edits logged in the activity log with field-level change details.
