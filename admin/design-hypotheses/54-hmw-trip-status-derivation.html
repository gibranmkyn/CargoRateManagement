HMW-54

How might we derive and display trip-level status from job statuses so ops planners can instantly see trip progress without expanding?

Client feedback: "Trip status should only be verified when all jobs are verified." Currently trips have no status column — status is implicit in Active/Completed filter tabs.

Derivation Logic (all options use this)

if ALL non-cancelled jobs are Verified → Trip = Verified
if ANY job is Completed & none Pending/In Progress → Trip = Completed
if ANY job is In Progress → Trip = In Progress
if ANY job is Pending → Trip = Pending
if ALL jobs are Cancelled → Trip = Cancelled
Option A

Trip Status Chip (worst-status derivation)

Add a Status column to the trip row. Trip inherits the lowest-progress non-cancelled job status. Same 5-color chip system as jobs. Consistent visual language across trips and jobs.

Pros

  • Same visual language as job-level status — consistent 5-color system
  • Instantly scannable — one chip tells you where the trip stands
  • Filter tabs (Active/Completed/Verified) align directly with chip colors

Cons

  • Single chip hides distribution (trip with 4 Verified + 1 Pending shows as Pending)
  • Might feel reductive for trips with many jobs in different stages
Option B

Job Status Summary (colored dot counts)

Show inline colored dots with counts on the trip row. Ops sees the full job status distribution at a glance without expanding. More information density, but denser to parse.

Pros

  • Full distribution visible — ops sees exactly how many jobs are in each stage
  • Answers "how far along is this trip?" and "what's blocking?" simultaneously
  • Trips with one straggler job are visually obvious (e.g., ●1 gray among ●4 green)

Cons

  • Harder to scan at speed — requires parsing multiple dots/numbers vs. single chip
  • Doesn't give a single "trip status" answer — ops must derive it mentally
  • Column gets wide with 4-5 different status groups
Option C

Verification Progress (fraction + bar)

Show "N/M verified" text with a thin progress bar. Answers the core question — "is this trip ready for billing?" — as a completion percentage. Optimized for the verification workflow.

▸ T00004 AliExpress 15 Apr 160-45678901 AliExpress WH → SZX Airport 65 bags

▸ T00005 Temu 13 Apr 160-56789012 Temu WH → SZX Airport 150 bags

Pros

  • Directly answers the client's core question: "is this trip fully verified?"
  • Visual progress bar makes near-complete trips pop (3/4 vs 0/4 is obvious)
  • Compact — fraction + bar fits in narrow column

Cons

  • Only tracks verification — hides whether jobs are Pending vs In Progress vs Completed
  • 0/4 could mean "all pending" or "all in progress" — no distinction
  • Progress bar is a new visual pattern not used elsewhere in the design system

Recommendation

Option A — Trip Status Chip is the strongest choice. It uses the existing 5-color status system (no new visual patterns), is instantly scannable at speed, and directly satisfies the client's request: trip shows as Verified only when all jobs are Verified.

Option B's micro-counts are more informative but violate the design principle of "every element should be parseable at a glance." Ops managing 30+ trips doesn't want to mentally sum ●1 ●2 ●1 — they want "is this trip done or not?" Option C is verification-centric but hides the Pending/In Progress/Completed distinction, which matters for daily ops (not just billing).

The derivation logic is simple: Trip status = worst(non-cancelled job statuses). Ordering: Pending < In Progress < Completed < Verified. If all cancelled, trip is Cancelled. This also updates the filter tabs — the existing Active/Completed split should evolve to match: Active (Pending + In Progress), Completed (all Completed or mixed Completed/Verified), Verified (all Verified).
