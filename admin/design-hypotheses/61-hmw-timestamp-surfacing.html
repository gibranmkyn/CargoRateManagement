HMW-61: Timestamp Surfacing on Status & Verification

"How might we show when a job or trip last changed state, without adding another colored element or crowding already dense rows?"
Context: Kim asked for timestamps on both job status and verification status. Full transition history already lives in the activity log — the question is what should sit on the row for at-a-glance scanning. Depends on HMW-60 (dual columns for Status + Verification) being resolved. Honors the "one dominant pill per row" rule — timestamps must render as neutral mono text, never colored pills.
Scoping decision. We show one timestamp per row per state column — when that state was last entered. The full transition sequence (Pending → In Progress → Completed → Verified, with every author and time) stays in the activity log. The highest-value question a planner asks while scanning is "how long has this been stuck here?", not "what was the exact path?" So: current-state timestamp on the row, everything else in the log.

Option A — Inline relative + tooltip absolute

Recommended
Small mono timestamp directly under each chip. Relative for <7 days ("2d ago"), absolute thereafter ("Apr 12"). Hovering the chip reveals a tooltip with the precise timestamp and actor. No new columns, no colored elements added. Row height grows by ~11px — similar to how we already stack trip ID + customer name in a single cell.

Trade-offs

A · Inline under chip

  • Per-state timestamp — Status and Verification answered independently
  • Answers "how long has this been stuck?" without a hover
  • No extra column; no extra color — mono gray text
  • Tooltip carries precise time + attribution
  • Row height grows ~11px (consistent with Trip/Customer stack)

B · "Updated" column

  • Sortable — true Excel-like "what changed last"
  • Row height unchanged
  • Can't tell if the update was a status change or verification change at a glance
  • +1 column on already-wide table (after HMW-60 added Verification)
  • Two separate filters but one combined timestamp — semantic mismatch

C · Tooltip only

  • Zero layout impact
  • Cleanest table
  • Requires hover to answer "how long" — breaks scan-flow
  • No signal on touch devices (less relevant but still)
  • Fails Kim's explicit "pls include timestamp" ask — she wants visible, not on-demand

Formatting conventions (apply to chosen option)

Relative for recent, absolute for older.
• just now — less than 1 minute
• 5m ago / 23m ago — under an hour
• 3h ago — under 24h
• 1d ago / 6d ago — under 7 days
• Apr 12 — 7+ days, same year
• Apr 12, 2025 — different year
Tooltip always shows absolute + attribution.
Format: Apr 20, 2026 · 14:32 · by Ops Admin
Attribution: "Ops Admin" for admin actions, vendor company name for vendor actions (matches activity log rule from CLAUDE.md).
Which timestamp do we store?
Job.statusChangedAt — when job status last transitioned. Updated on Pending→In Progress, In Progress→Completed, or reset on re-work.
Job.verificationChangedAt — when verification status last transitioned. Updated on Pending→Verified, Pending→Rejected, Verified→Pending (unverify), Rejected→Pending (vendor re-upload).
Both are plain ISO strings; the activity log remains the source of truth for full history.

Verdict — PENDING (awaiting user decision)

Recommendation: Option A (Inline under chip + tooltip absolute). The mono gray text under each chip is neutral metadata — no color competition with the Status pill. Answers Kim's visible-timestamp ask without adding a column, and distinguishes status-age from verification-age because each column owns its own timestamp. Tooltip carries the precise time + attribution for audits. Activity log remains authoritative for full transition history.

Conventions: relative under 7 days, absolute older. Tooltip always shows absolute + attribution. Two underlying fields: statusChangedAt and verificationChangedAt. Applies to both Jobs and Trips tables — HMW-63 will inherit.
