HMW-60: Verification Column & Reject UX

"How might we surface verification as a first-class state on the Jobs table — separate from job status — while keeping the reject action discoverable without crowding the row?"
Context (from Kim's feedback, Apr 2026):
• Job Status (Pending / In Progress / Completed / Cancelled) = vendor-driven lifecycle. Triggered by Start Job + proof upload.
• Verification Status (Pending / Verified / Rejected) = admin-driven sign-off on vendor's work. Independent of job status.
• A Completed job can be Pending verification, Verified, or Rejected — these are two parallel states, not one combined state.
• Reject is per-job (not per-document, simplified 2026-04-20). Rejection requires a reason. Vendor re-uploads any proofs and verification flips back to Pending. No round limit.
• Unverify ≠ Reject. Unverify is admin self-correction (no reason, keeps proofs). Reject is admin flagging bad vendor work (reason required).
What this HMW decides. (1) How job status and verification status coexist on the Jobs table without blowing up row density. (2) Where the Reject / Verify / Unverify actions live. (3) How rejection reason is captured. Timestamp placement is out of scope — deferred to HMW-61 so this mockup doesn't try to solve two problems at once.

Option A — Dual Columns

Recommended
One column for Status, one for Verification. Each state gets its own cell, its own filter, its own scan lane. Verification column is empty (em-dash) until a job reaches Completed — visually reinforcing that verification only begins once the vendor has done their part.

Trade-offs

A · Dual Columns

  • Two filters map to two columns — mental model is clean
  • Verification column only populates at Completed+ — visual rhythm teaches the lifecycle
  • Sortable and scannable per axis
  • +1 column on an already dense table
  • Rejection + verification share visual weight with status

B · Stacked Pill

  • Column count unchanged
  • Groups related info spatially
  • Row height grows — breaks Excel-density feel
  • Two semantics in one cell; harder to filter/sort
  • Verification tag disappears for non-Completed rows → visual inconsistency

C · Absorbed Status

  • Fewest pixels spent
  • Matches current prototype — zero migration cost
  • Conflates job lifecycle with admin sign-off — exactly what Kim flagged
  • "Completed" is ambiguous: awaiting review? unreviewed? nobody's looked?
  • No place for separate Verification filter without adding column back

Reject / Verify / Unverify — action surface

Actions live in the slide-out, not the row. The Jobs row remains scan-only. Clicking any row opens the existing slide-out (HMW-49 status action bar pattern). The action bar at the top adapts based on current status + verification state. Three scenarios shown below.

Unverify vs Reject — different semantics.
  • Unverify = admin self-correction. No reason required. Keeps all proofs. Flips verification back to Pending so admin can edit fees/quantities or re-check. No vendor notification.
  • Reject = admin flags vendor's work as insufficient. Reason required. Vendor sees the reason and is expected to re-upload. Verification becomes Rejected (not Pending) so the admin knows "waiting on vendor," not "waiting on me."

Vendor side — how a rejected job appears

Vendor opens their Job Detail page. A rejection callout sits near the top, above proofs. The rejection reason is the first thing they see; the re-upload affordance is one click away. When they upload any new proof, verification flips back to Pending automatically and the callout disappears.

Verdict — PENDING (awaiting user decision)

Recommendation: Option A (Dual Columns). It's the only option that matches Kim's conceptual ask — separating job status (vendor) from verification status (admin). The +1 column cost is real but bounded: the verification column is narrow (~100px) and mostly empty em-dashes until Completed. Options B and C either blur the separation or add row height, both of which work against the Excel-density aesthetic.

Reject UX: inline reason capture in the slide-out (not a separate modal), consistent with the HMW-51 cancel/replace pattern. Unverify kept as a distinct button with no reason input. Vendor sees a red callout with reason on re-upload.

Open for HMW-61: where timestamps attach to status and verification chips — inline? tooltip? attribution?
