HMW-50: Create Shipment — Minimize Vendor Clicks

"How might we minimize clicks when creating a shipment, given that 80% of shipments assign all jobs to the same vendor?"
The problem: Currently, adding 4 jobs to a shipment requires selecting the vendor 4 separate times from a dropdown — even when all 4 go to the same vendor (which happens 80% of the time). For ops creating 10-50 shipments/day, this adds up to hundreds of redundant clicks.

Current flow (4 jobs, same vendor): Click +FM → select HaleSun. Click +EC → select HaleSun. Click +CS → select HaleSun. Click +OH → select HaleSun. = 12 interactions (4 pill clicks + 4 dropdown opens + 4 option selects).

Constraint: The 20% of shipments that need different vendors per service must still be easy. Don't break the minority case to optimize the majority.

Option A: Default Vendor

Recommended
Thesis: Add a "Assign to" vendor dropdown above the service pills. When you click a service pill, the new job is created with that vendor pre-filled. For mixed-vendor shipments, override the vendor in the individual job card. One new element, zero new concepts.
Pros
• Minimal change — adds one dropdown, keeps existing service pills
• 80% case: pick vendor once → click services → done (6 clicks vs 12)
• 20% case: override per-job dropdown works exactly like before
• Zero learning curve — dropdown + pills are familiar patterns
Cons
• Still requires 4 individual pill clicks for 4 services
• The per-job vendor dropdown is still there (for overrides) — could feel redundant
• Default vendor state might confuse: "why is HaleSun selected already?"

Option B: Vendor + Multi-Select Services

Fewest clicks
Thesis: Replace the current "click pills one-by-one" with a combined interface: select a vendor, then toggle which services they'll handle using checkboxes. An "All" button toggles everything on. Click "Add Jobs" once to create all jobs at once. For mixed vendors: add jobs for vendor A, change vendor, add jobs for vendor B.
Pros
• Fewest clicks: vendor (2) + "All" (1) + "Add Jobs" (1) = 4 clicks for 4+ jobs
• Natural mental model: "HaleSun handles FM, EC, CS, OH" = one thought, one action
• "All" button makes the 80% case a single click after vendor selection
• Mixed vendors: just change vendor and add more services
Cons
• Bigger redesign of Step 2 — new interaction pattern
• "Add Jobs" button is a batch action (less immediate than current pills)
• Need to handle edge case: what if you add FM for HaleSun then try to add FM for Gonda?

Option C: Auto-Fill from Last Vendor

Zero new UI
Thesis: No new UI elements. When you select a vendor for one job, all subsequently added jobs auto-fill with that same vendor. The first job requires manual vendor selection; jobs 2-4 inherit it. You can always change per-job. The system learns from your behavior within the session.
Pros
• Zero new UI elements — uses the existing job cards and dropdowns
• Works naturally: select vendor once, add more services, they inherit
• No new interaction patterns to learn
• Degrades gracefully: if you change vendors, the next job inherits the latest
Cons
• Not visible — user doesn't know auto-fill will happen until it does
• "Magic" behavior can surprise: "why is HaleSun already selected?"
• Doesn't reduce clicks as much: still 4 pill clicks + 1 vendor selection = 6 clicks
• For the first shipment of the day, there's no "last vendor" to auto-fill

Recommendation: Option A (Default Vendor) with cherry-pick from B

Option A is the right base — add a "Assign to" vendor dropdown above the service pills. It's minimal change, zero new concepts, and cuts the 80% case from 12 interactions to 5-6. The existing service pills stay (they're proven good for service-first thinking).

Cherry-pick from B: Add a "+ All" button after the service pills. One click creates all 5 services at once with the default vendor. This brings the 80% case down to: vendor (2 clicks) + "All" (1 click) = 3 clicks total to create a 5-job shipment. For selective, ops clicks individual service pills as before.

Skip Option C — invisible auto-fill is magic behavior that surprises users. The "Assign to" dropdown in Option A is explicit and discoverable.

The 20% case (mixed vendors): Set default to vendor A → add their services → change default to vendor B → add their services. Or override per-job with the "change" link. Both work naturally.

Click count comparison:
• Current: 12 interactions (4 pill clicks + 4 dropdown opens + 4 option selects)
• Option A: 5 interactions (1 vendor + 4 pills)
• Option A + "All": 3 interactions (1 vendor + 1 "All")
• Option B: 4 interactions (1 vendor + "All" + "Add Jobs") — similar but bigger redesign
• Option C: 6 interactions (1 vendor + 4 pills, but auto-fill is invisible)
