H06: Action Feedback

"How might we give ops instant confidence their action worked — without interrupting their flow?"
Scenario

The ops planner just created a new trip for AliExpress, changed 3 job statuses from "Pending" to "In Transit", and uploaded a proof of delivery document — all within 2 minutes. They need confirmation that each action was saved successfully, without having to stop and verify. If they don't trust the system saved their work, they'll double-check everything manually, slowing them down.

Option A: Toast Notifications

Industry standard — familiar pattern
How it works: Every action triggers a toast notification in the top-right corner. Green for success, amber for warnings, red for errors. Toasts auto-dismiss after 4 seconds. The planner sees text confirmation ("Trip created — TRIP20260325-005") without needing to click or navigate anywhere.
Pros
Industry standard — every SaaS tool uses this pattern
Non-blocking — planner can keep working while the toast shows
Clear text confirmation of exactly what happened
Color-coded severity (green/amber/red) provides quick scan
Cons
Can be missed if planner is looking at a different part of the screen
No persistence — gone after 4 seconds, no way to review
Rapid-fire actions = stacked toasts that feel noisy
Toast position (top-right) is far from the action area (center/left)

Option B: Inline State Changes

Spatial feedback — co-located with the action
How it works: No toast. Instead, the changed element itself provides feedback through visual animation. A new trip row gets a green left border glow and a "JUST CREATED" tag that fades after 10 seconds. A status change causes the chip to pulse briefly. An uploaded proof file appears with a "NEW" badge. Feedback is always right where the action happened.
Pros
Feedback is spatially co-located — impossible to miss if you're looking at the element
No extra UI chrome (no toast layer, no notification stack)
Feels "native" — like the data itself is responding to your action
Scales naturally — each element handles its own feedback
Cons
If the user scrolled away from the element, they see nothing
No text confirmation of what happened — just visual change
Animations can be subtle and missed on large screens
"JUST CREATED" tag needs to auto-remove, adding complexity

Option C: Toast + Inline Combined

Recommended — redundant feedback for zero doubt
How it works: Combines both approaches. Every action shows a toast notification AND the affected element provides inline visual feedback. The toast gives text confirmation ("Trip created"), the inline highlight gives spatial confirmation (green glow on the row). For rapid-fire actions (e.g., changing 5 statuses in a row), toasts are batched: "3 jobs updated" instead of 3 separate toasts.
Pros
Redundant feedback = zero chance of missing it
Toast handles "what happened" (text), inline handles "where" (spatial)
Batch toasts solve the stacking problem for rapid actions
Users who prefer toast get toast; users who prefer inline get inline
Cons
Potentially noisy if many actions happen fast
More implementation work — two systems to maintain
Need smart batching logic to avoid overwhelming the planner

Recommendation: Option C — Toast + Inline Combined

Dual-channel feedback gives the planner maximum confidence with minimal disruption.

  • Toast = text confirmation. "Trip created — TRIP20260325-005" gives the planner words they can read. If they're looking at the top-right, they see it immediately.
  • Inline = spatial confirmation. The green glow and "JUST CREATED" tag confirm the action right where it happened. If the planner is looking at the table, they see it immediately.
  • Batch toasts solve the noise problem. If the planner changes 5 statuses in 10 seconds, one toast says "5 jobs updated to In Transit" instead of 5 separate toasts. Each row still gets its individual pulse animation.
  • Error handling is toast-first. If an action fails, the toast turns red with a clear error message and a "Retry" button. The inline element reverts to its previous state.
  • Toast auto-dismiss: 4s for success, sticky for errors. Success toasts disappear on their own. Error toasts stay until dismissed or retried.
