HMW: Rate Anomaly & Billing

"How might we help billing admin catch rate discrepancies between agreed vendor rates and actual invoices — before approving payment?"
Current problem: Vendor invoices sometimes don't match agreed rates. Gonda invoices RM 180 for EC when the agreed rate is RM 150. HaleSun invoices RM 500 for FM when the rate card says RM 450. Currently, billing admin manually checks each invoice line against a spreadsheet.

Scale: Teleport processes ~200 delivery orders per month, each with 3-5 jobs. That's 600-1000 invoice line items to validate. Manual checking takes ~2 hours per day. With rate cards now in the system, we can automate the comparison and flag anomalies instantly.

Impact: Rate discrepancies average RM 15-30 per flagged job. At ~10% flag rate, that's 60-100 flagged jobs saving RM 900-3,000/month in overcharges caught.

Option A: Inline Variance in Validation Page

Lightweight
How it works: The existing validation page (from the billing flow) gains new columns: "Agreed Rate", "Invoice Amount", and "Variance". For each job, the system shows the rate from the rate card. Admin enters the vendor's invoiced amount. The system instantly calculates variance and color-codes: green = match, red = over, blue = under, amber = no rate on file. Admin reviews flagged lines and approves or disputes.
Pros
- Integrates with existing validation workflow
- Simple — just extra columns in a table admin already uses
- Variance is calculated automatically
- Clear visual hierarchy: flagged rows stand out
Cons
- Admin still enters invoice amounts manually per job
- No invoice document view alongside the comparison
- Works per order, not per vendor (can't see all Gonda overcharges at once)
- No historical trend ("Gonda has been overcharging EC for 3 months")

Option B: Invoice Upload + Side-by-Side View

Full solution
How it works: Admin uploads the vendor invoice (PDF or image) and it displays on the left side of a split view. The right side shows the agreed rates from the rate card. Admin reads the invoice and enters actual amounts — the system highlights discrepancies. The invoice stays visible for reference while admin works through each line.
Pros
- Invoice and rate card side by side — less context switching
- Admin can reference the actual document while entering amounts
- Professional workflow — similar to accounting software
- Could support invoice OCR in the future
Cons
- Invoice parsing/OCR is hard (different formats per vendor)
- Still requires manual amount entry for Phase 1
- Split view uses more screen space
- Upload workflow adds a step before validation can begin

Option C: Invoice Amount Field Per Job

Recommended
How it works: Add an "Invoice Amount" field to each job, editable by billing admin in the validation workflow. When admin enters the amount, the system auto-compares against the agreed rate and shows variance inline: RM 0 (match), +RM 30 (over), -RM 20 (under). The sub-table gains "Invoice" and "Variance" columns visible only in validation mode. No separate page needed — the comparison lives right in the existing order detail view.
Pros
- Lightweight — just one new field per job
- Lives in the existing sub-table, no new page
- Variance calculated automatically per job
- Admin works in the same context as the order data
- Easy to implement — add field + calculation + styling
Cons
- Admin enters amounts manually (no invoice upload)
- No side-by-side invoice view
- Validation happens per order, not per vendor batch
- Per-kg/per-bag rates need quantity-aware variance calculation

Recommendation: Option C — Invoice Amount Field Per Job

Why Option C: It's the lightest implementation that gives immediate value. One new field per job ("Invoice Amount"), plus automatic variance calculation. No new pages needed — the comparison lives right in the sub-table, visible in the billing admin's validation workflow. Admin enters the amount, the system does the math, red flags appear instantly.

Unit-aware variance: For per-bag and per-kg rates, the system compares the total calculated cost (rate x quantity) against the invoiced amount — not the unit rate itself. This handles volume-based services correctly. Example: agreed RM 25/bag x 42 bags = RM 1,050 agreed, invoiced RM 1,100 = +RM 50 variance.

Three variance states: Green "Match" (RM 0 variance), Red "Over" (vendor charged more than agreed), Blue "Under" (vendor charged less — possibly a credit or updated rate). Amber "No rate" when no rate card exists for this vendor/service/location — a signal to add one.

Future: anomaly summary. Once we have a few months of data, surface the recurring overcharges in a summary view (shown in the last mockup). This turns one-off disputes into a pattern: "Gonda has overcharged EC at Shekou Port 5 times this month — renegotiate or update the rate card."

Data model: Add invoiceAmount?: number and varianceStatus?: "match" | "over" | "under" | "no_rate" to the Job interface. The variance is calculated on entry, not stored — it's always invoiceAmount - agreedCost.
