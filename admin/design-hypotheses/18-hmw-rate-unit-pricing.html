HMW: Rate Unit Pricing

"How might we model vendor rates that reflect real logistics pricing — where some services charge flat, some per-kg, some per-bag — without overcomplicating the UI?"
Current assumption: The initial rate model (HMW-16) assumes a flat rate per job — e.g., RM 450 for FM Trucking from A to B. This works for fixed-price services.

Real pricing: Vendor contracts use mixed units:
• FM Trucking — flat per trip (RM 450)
• CR Cargo Retrieval — per bag (RM 25/bag)
• OH Origin Handling — per kg (RM 12/kg)
• EC Export Customs — flat per declaration (RM 150)
• CS Customs Submission — per CBM for large shipments (RM 80/CBM)

Impact: If we only support flat rates, cost estimates for volume-based services will be wrong. A 50-bag order at RM 25/bag = RM 1,250, not a flat RM 25. Billing reconciliation breaks if the unit type is ignored.

Option A: Flat Rate Only (Simplest)

Current baseline
How it works: Every rate is a single number — no unit type, no calculation. RM 450 for FM from A to B, RM 150 for EC at a location. Cost = rate, regardless of how many bags or how much weight. The rate card table just has a "Rate" column.
Pros
- Simplest possible data model
- No calculation logic needed
- Easy to understand and maintain
- Good enough for fixed-price services (FM, EC)
Cons
- Inaccurate for volume-based services (CR per-bag, OH per-kg)
- Cost estimates wrong for large or small shipments
- Doesn't reflect actual vendor contracts
- Billing reconciliation will still have discrepancies

Option B: Rate + Unit Type

Recommended
How it works: Each rate has a unit type: flat, per-kg, per-bag, or per-CBM. The rate card shows both the number and the unit: "RM 450 flat" vs "RM 12 /kg" vs "RM 25 /bag". Cost calculation uses the unit: flat → rate, per-kg → rate × weight, per-bag → rate × bags, per-CBM → rate × volume. The create form shows the calculated cost with a transparent breakdown.
Pros
- Accurate cost estimates for all service types
- Matches real vendor contracts
- Billing reconciliation can compare apples to apples
- Transparent — ops sees rate, unit, quantity, and calculated cost
Cons
- More complex data model (unit type field, calculation logic)
- Ops needs to understand that some rates are per-unit
- Order total depends on bag count and weight being entered accurately
- Need to handle edge case: what if bags/weight not yet known?

Option C: Tiered Pricing

Future / overkill
How it works: Rates have volume tiers: 0-500 kg = RM 14/kg, 501-1000 kg = RM 12/kg, 1001+ kg = RM 10/kg. The system picks the tier based on the shipment's weight/bags, then calculates cost. Supports progressive tiers (different rate per bracket) or slab tiers (single rate based on total).
Pros
- Most accurate for large-volume contracts with volume discounts
- Reflects how enterprise logistics contracts actually work
- Incentivizes ops to consolidate shipments for better rates
Cons
- Way too complex for Phase 1 — most vendors use simple per-unit pricing
- Data entry burden: admin enters multiple tier rows per rate
- Tier edge cases (split shipments, partial weights) add complexity
- Only 1-2 vendors currently have tiered contracts

Compare: flat Option B at RM 10.50/kg would be RM 19,320. Tiered pricing is more nuanced but significantly harder to maintain.

Recommendation: Option B — Rate + Unit Type

Why Option B: It's the minimum viable accuracy. Flat rates (Option A) are wrong for per-bag and per-kg services — the cost difference can be 10x. Unit types add one field to the data model but make every cost estimate correct. Tiered pricing (Option C) is overkill — only 1-2 vendors use tiers, and we can handle those as manual overrides for now.

UI pattern: Show the unit type as a small badge next to the rate: RM 25.00 /bag vs RM 450.00 flat. In the create form, show the full calculation: rate × quantity = cost. This makes it transparent and easy to verify.

Edge case: If bags or weight aren't entered yet, show "RM 25.00 /bag × ? bags" with a placeholder. The total updates live as ops fills in the shipment details. This encourages ops to enter accurate quantities.

Supported units for Phase 1: flat, per-kg, per-bag. Add per-cbm when a vendor contract requires it.
