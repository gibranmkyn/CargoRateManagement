HMW-33: Sparse Grids — L2 Fee Applicability

How might we handle L2 fees that don't apply to every route — should the rate sheet show all 20 rows or only the applicable ones?
The problem: There are 20 L2 fee types across 5 L1 services. A vendor like HaleSun only has rates configured for 5 of those 20 fees (FM001 Warehouse Transfer, FM002 Pickup, FM003 Cross-Border Handling, CR001 Registration, CR002 Cargo Retrieval Pick Up). The other 15 rows would show nothing but dashes across every route column.

Why this matters: FM003 (Cross-Border Handling) only applies to cross-border routes. CS004 (Security X-Ray Screening) only applies to certain airports. If we always show all 20 rows, most of the grid is empty — visually noisy, confusing, and makes the configured rates harder to find. But hiding rows means the user might not realize a fee type exists.

Real scenario: HaleSun has 5 configured L2 fees across 3 routes = 15 cells with data. A full 20-row grid across 3 routes = 60 cells total — 75% of the grid is empty dashes. That's a bad signal-to-noise ratio for a tool that's supposed to feel like a "smart spreadsheet."

Option A: Show Only Configured L2 Fees (Toggle to Expand)

Recommended
Thesis: The rate sheet only shows L2 rows where the vendor has at least 1 rate configured across any route. A "Show all L2 fees" toggle at the bottom reveals the full 20-row grid for adding new rates. Clean by default, complete when needed.

How it works:

  • Default view: only L2 rows with at least 1 configured rate are visible
  • L1 group headers only appear if that group has visible L2 rows
  • Footer shows "5 of 20 L2 fees shown" with a toggle link
  • Toggle "Show all L2 fees" expands to the full 20-row grid — unconfigured rows have empty input cells ready for entry
  • Adding a rate to a previously hidden row keeps it visible after collapsing
Pros
- Clean default: only signal, no noise
- 5 rows vs 20 — dramatically less scrolling
- L1 groups that have no rates don't clutter the view
- Toggle preserves full discoverability
- Matches the "smart spreadsheet" ethos — show what matters
Cons
- Users must know to click toggle to add new fee types
- Slight learning curve: "where are the other fees?"
- Toggle state needs to persist per vendor or reset on switch

Option B: Show All 20 Rows, Gray Out Empty

Full Grid
Thesis: Always show all 20 L2 fee rows in the rate sheet. Rows with no rates configured get a subtle gray background and muted text, making them visually recede but still scannable. No toggle — what you see is what you get.

How it works:

  • All 20 rows always visible, grouped by L1 service
  • Rows with configured rates: white background, dark text, editable inputs
  • Rows with no rates: #fafafa background, muted text, empty inputs with dashed borders
  • User can click into any empty cell and start typing to add a rate
Pros
- Full picture always visible — no hidden state
- Zero learning curve — every fee type is right there
- Easy to add new rates — just click and type
- No toggle to discover or manage
Cons
- 75% of grid is empty dashes — very noisy
- Configured rates get lost in the sea of empty rows
- 20 rows + 5 L1 headers = 25 rows to scan even for 5 configured fees
- Feels like an incomplete spreadsheet, not a "smart" one

Option C: Applicable Per Route (Dynamic Rows)

Radical
Thesis: Different route columns show different L2 fee rows. FM003 (Cross-Border Handling) only appears in columns for cross-border routes. CS004 (Security X-Ray) only appears for airport location columns. The grid rows vary per column — the most accurate representation, but the most confusing.

How it works:

  • Each route column is tagged with attributes (cross-border: yes/no, airport: yes/no, etc.)
  • L2 fee types have applicability rules — FM003 is tagged "cross-border only"
  • Grid only shows cells where the fee type is applicable to the route
  • Non-applicable cells show a gray "N/A" instead of a dash or empty input
Pros
- Most accurate — only shows fees that can actually apply
- Eliminates confusion about "should I fill this cell?"
- Makes route-specific fees obvious
Cons
- Breaks the spreadsheet mental model — rows vary per column
- Very confusing to scan: "why does this row have cells in some columns but not others?"
- Complex to implement — need applicability rules per L2 fee
- Hard to add new routes — must define applicability first
- Users expect grids to be rectangular, not ragged

Verdict: Option A — Show Configured Only, Toggle to Expand

Recommended because: The rate sheet's job is to show what's configured, not what's possible. A vendor with 5 fees shouldn't wade through 15 empty rows to find them. The "Show all" toggle preserves full discoverability for when you need to add new fee types, but the default view is clean and focused.

  • Default view: Only L2 rows with at least 1 configured rate — 5 rows for HaleSun instead of 20
  • L1 groups hidden if empty: If a vendor has no CS, EC, or OH rates, those groups don't appear
  • "Show all 20 L2 fees" toggle: Reveals full grid with empty rows in muted style for adding new rates
  • Unconfigured rows: Gray background (#fafafa), muted text, empty inputs with subtle borders — clearly secondary
  • Footer summary: "5 of 20 L2 fees configured" gives context without requiring expansion
  • No applicability rules needed: Unlike Option C, we don't need to define which fees apply to which routes. A dash means "not configured" — the user decides if that's intentional or an oversight

Why not Option B: Full grid is visually noisy — 75% empty cells makes it hard to scan. The gray styling helps, but the fundamental problem is signal-to-noise ratio.
Why not Option C: Dynamic rows per column breaks the spreadsheet mental model. Users expect grids to be rectangular. N/A cells create confusion about what's allowed vs. what's simply not configured.
