HMW-63: Trips Page — Status & Verification Columns

"How might we add trip-level Status and Verification to the Trips table — simply enough that ops planners understand at a glance, detail lives one click away?"
Design principle for this mockup: simple design users immediately understand beats clever design that preserves more signals. Trip row answers one question per column: is this trip done? and is this trip billable? Everything else — progress counts, rejected job counts, per-job timestamps — lives inside the expanded view, where it already belongs.
Locked decisions:
• Trip Status auto-derived: In Progress if any job in progress; Completed if all non-cancelled jobs completed; Pending otherwise. No Verified in trip status.
• Trip Verification auto-derived binary: Verified if all non-cancelled jobs verified; Pending otherwise. Rejected jobs don't bubble up — trip stays Pending.
• Filters on Trips page are trip-level (status + verification).
• Expanded sub-rows inherit HMW-60/61 pattern unchanged (job chips + inline timestamps + tooltip attribution).

The design

Two new columns at the end of the Trips table. Status — filled pill (Pending / In Progress / Completed). Verification — text-only tag (Pending / Verified). No timestamps on trip rows, no progress counts, no per-row rejected callouts. A trip row gives you the answer; the expanded view gives you the story.
Rejected jobs are still findable at a glance — two lightweight surfaces, neither adds a new column or sub-stat:
  1. Row tint. Any trip containing at least one job with verificationStatus === 'Rejected' gets a subtle red background tint (same .rj pattern the Jobs table already uses). No icon, no text, no count — just "this row needs attention."
  2. Filter chip. A "Has Rejected" shortcut chip in the filter bar jumps straight to those trips. Not a verification state (verification remains binary), just a practical filter.

What we considered and rejected

× Timestamps on trip rows.

A trip-level timestamp is ambiguous — "Completed · 3h ago" really means "the last job to complete did so 3h ago," which you can just as well see by expanding the trip. Trip rows stay clean; job rows carry the time info where it's unambiguous.

× "N/M verified" progress sub-stat.

Tempting because it preserves HMW-54's billing-readiness signal, but it adds a second line of mono text to every row and forces the reader to do arithmetic. "Pending" already tells you billing isn't ready; if you want to know how close, expand the trip and you see each job's state directly.

× Rejected-job count as red mono text on the row.

Adds a number to every rejected-containing row and forces the reader to interpret it. The subtle red row tint + "Has Rejected" filter chip answer the same need — "which trips have rejects?" — without making you read. Exact count lives one click away, in the expanded sub-table.

× HMW-54's progress bar.

Used a green/amber fill to encode completion percent. Re-introduces secondary color and diverges from the Jobs-page visual language for the same concept. The binary pill is the shared vocabulary now.

Derivation rules

Trip Status (computed on read, no stored field)
  • In Progress if any non-cancelled job is In Progress
  • Completed if all non-cancelled jobs are Completed (regardless of verification state)
  • Pending otherwise
  • Cancelled jobs are excluded (a trip of 3 jobs where 1 is cancelled + 2 completed = Trip Completed)
Trip Verification (computed on read, no stored field)
  • Verified if all non-cancelled jobs have verificationStatus === 'Verified'
  • Pending in every other case (any job pending, any job rejected, trip still in progress)

Verdict — PENDING

Trip row = one question per column: is this trip done? (Status pill) and is this trip billable? (Verification tag). All secondary detail — per-job state, timestamps, rejected jobs, progress counts — lives in the expanded sub-rows, which already follow HMW-60 / HMW-61 conventions. No new vocabulary, no arithmetic, no colored progress bars.
