H07: Trip Completion & Billing Readiness

"How might we help ops admins confidently know which trips are ready for vendor payment — and which aren't?"

The end goal of every trip is billing. Teleport pays vendors for completed services. The admin needs to know: (1) Are all jobs done? (2) Is proof uploaded for every job? (3) Can I confidently validate this trip for payment?

The cost of getting this wrong: Paying a vendor for incomplete work, or delaying payment for completed work. Both erode vendor relationships.

Option A: Visual Indicator on Trip Row

Passive — information at a glance
How it works: When ALL jobs in a trip are "Completed" AND ALL jobs have at least 1 proof document uploaded, the trip row shows a "Ready for Billing" badge. The admin scans the list and sees which trips are ready without clicking into anything.
✓ Pros
• Zero-click — visible in the trip list
• Clear binary state: ready or not ready
• Works with the existing table layout
• Simple to implement
✕ Cons
• No action — just information. Admin still needs to navigate somewhere to validate.
• Doesn't show WHAT's missing if not ready
• Can clutter the trip row if there are many indicators

Option B: Billing Readiness Checklist in Expanded View

Active — shows what's missing + action
How it works: When a trip is expanded, show a "Billing Readiness" section below the job cards. It shows a checklist: each job needs ✓ Completed status + ✓ Proof uploaded. Items that are done show green checks; items that aren't show what's missing. A "Validate for Billing" button activates only when all checks pass.
✓ Pros
• Shows exactly WHAT's missing — actionable
• "Validate" button is the clear next step
• Progress bar gives a visual sense of how close the trip is
• Naturally fits in the expanded trip view
✕ Cons
• Requires expanding the trip to see readiness
• Adds vertical space to the expanded view
• "Validate" action needs careful design to prevent accidental validation

Option C: Separate "Validation" View

Dedicated workflow for billing admins
How it works: The "Validation" nav link leads to a dedicated page showing ONLY trips that are candidates for billing (all jobs completed). Each trip shows a summary with proof status, checklist, and a validate button. This is the admin's workspace — separate from the ops planner's trip list.
✓ Pros
• Clean separation: ops manages trips, admin validates for billing
• No clutter in the trip list — billing indicators are in their own space
• Can add billing-specific features (cost breakdown, invoice generation) without polluting the trip list
• Matches the CR requirement for a "validation dashboard"
✕ Cons
• Admin needs to switch between trips list and validation view
• Duplicates trip information in a second view
• More pages to build and maintain

Recommendation: Option B + C combined

  • Option B (checklist in expanded view) gives ops planners immediate visibility into what's missing — they can fix it right there (upload proof, change status).
  • Option C (validation page) gives the billing admin a dedicated workspace — they don't want to scroll through all trips, just the ones ready for payment.
  • Combined: The trip list shows a small readiness indicator (from Option A) so ops knows at a glance. The expanded view shows the checklist (Option B). The Validation page (Option C) is the admin's billing workspace.
  • This matches the CR requirement which explicitly mentions a "Trips Validation dashboard" for admins.
