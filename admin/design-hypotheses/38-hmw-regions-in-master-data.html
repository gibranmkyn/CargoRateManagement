HMW-38: China Regions in Master Data

How might we integrate the standardized China city/district hierarchy into the Master Data module so it connects cleanly to both FTL trucking rates (district-to-district) and facility-based service rates?
Two address systems coexist:

1. Facilities (current Locations tab): Specific places like "TikTok WH, Shenzhen" or "SZX Airport Cargo Terminal". Used by EC, CS, CR, OH services. A facility sits WITHIN a district.

2. Districts (new China Regions data): Standardized admin divisions like "宝安区, 深圳" or "洪梅镇, 东莞". Used by FM Trucking for FTL pricing. A district contains many facilities.

The relationship: Facility ∈ District ∈ City ∈ Province. TikTok WH is in 洪梅镇 which is in 东莞市 which is in 广东省. The FTL rate from 洪梅镇 → 宝安区 covers trucking from TikTok WH to Shenzhen Airport.

The question: How should the Locations tab, Regions data, and rate cards connect in the Master Data UI?

Option A: Two Tabs — Facilities + Regions

Recommended
Thesis: Rename "Locations" to "Facilities" and add a new "Regions" tab showing the China city/district hierarchy. Each facility gets a districtCode field linking it to a region. FTL rates use district codes; service rates use facility IDs. Both live in Master Data but serve different purposes.

Master Data tabs become:

Facilities Regions Vendors Customers Services
  • Facilities tab (renamed from Locations): Warehouses, airports, ports, hubs. Each now has an optional districtCode field linking to the Regions hierarchy. The existing zone/type fields stay.
  • Regions tab: Read-only view of the China city/district hierarchy. Expandable tree: Province → City → Districts. Shows GB/T 2260 codes, bilingual names, and counts (facilities in district, FTL routes using this district).
  • Connection: When creating an FTL job, ops picks origin district + destination district (from Regions). The specific pickup/delivery facility is a detail within the order, not the rate lookup key.
✓ Pros
• Clean separation: facilities = where stuff happens, regions = standardized addresses
• FTL rate lookup by district code is unambiguous
• Facilities can be linked to districts for reporting
• Regions data is read-only (from GB/T standard) — no user editing needed
• Each tab does one job
✗ Cons
• One more tab in Master Data
• User needs to understand the facility → district relationship
• Linking facilities to districts requires a new field

Option B: Merge — Location = District + Facility in One Tab

Unified
Thesis: No separate Regions tab. Instead, the Locations tab shows a tree: City → District → Facilities. Districts are auto-populated from the China regions dataset. Facilities sit under their district. Everything in one view.
✓ Pros
• One tab for everything location-related
• The hierarchy is visible in one tree
• No separate linking step
✗ Cons
• 95 districts + facilities = very long tree
• Mixing standardized data (districts) with user data (facilities)
• District tree is read-only but facility rows are editable — confusing

Verdict: Option A — Two Tabs (Facilities + Regions)

Recommended because: Facilities and regions serve different purposes. Facilities are user-managed operational locations. Regions are a standardized reference dataset from the government. Mixing them in one view creates confusion about what's editable and what's not. Two tabs keeps each concept clean.

  • Regions tab: Read-only tree. Province → City → District. Shows GB/T codes, bilingual names. Counts of facilities and FTL routes per district.
  • Facilities tab: Existing CRUD table. New column: "District" showing the linked region code + name. Optional — not all facilities need a district link.
  • FM Trucking rates: Use district codes for origin/destination (from Regions)
  • EC/CS/CR/OH rates: Use facility IDs for location (from Facilities)
  • Create order: For FM jobs, ops picks origin district + destination district. For other jobs, ops picks a facility.
  • Rename: "Locations" tab → "Facilities" tab to avoid confusion with "Regions"
