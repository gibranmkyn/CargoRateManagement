HMW-52: Trips Table Column Restructure

How might we restructure the Trips table columns so the trip ID is the primary scannable identifier, dates and MAWB are immediately visible, and redundant columns are eliminated?
Client feedback: "Trip Number should be the leftmost column. Column order should be Trip Number, Pickup Date, Delivery Date, MAWB, then Route. The Jobs column feels redundant."

Current state: The table leads with Customer name, trip ID is a small gray chip underneath. The Jobs column shows colored vendor pills (e.g., "HaleSun · FM") taking ~20% of table width. The same job detail is available in the expanded sub-row.

Key question: Is the Jobs column truly redundant, or does it provide at-a-glance status value that the expansion doesn't? And does leading with Trip ID work when the Customer context matters for quick scanning?

Option A: Client's Column Order (Trip-First, No Jobs)

Client Request
Exactly what the client asked for: Trip Number leads, followed by dates and MAWB, then Route. Jobs column removed entirely. Customer name moves to a secondary position after the trip ID.
Pros:
• Trip ID is the primary lookup key ops uses in calls/WhatsApp
• Dates front-and-center matches planning workflow (what's moving today?)
• Removing Jobs column gives more room to other columns
• Cleaner, less visual noise per row
Cons:
• No job status visibility without expanding every row
• "Is this trip healthy?" requires a click to find out
• Loses the quick "3 jobs, 2 green, 1 red" signal
• Customer name is secondary — harder to scan by client

Option B: Trip-First + Compact Job Summary

Recommended
Trip ID leads (client's ask). Dates and MAWB follow. But instead of removing the Jobs column entirely, replace the verbose pills with a compact count + status summary. This preserves the "is this trip healthy?" signal in a fraction of the space.
Pros:
• Trip ID leads — matches ops lookup workflow
• Dates visible for planning — "what's moving today?"
• Job health at a glance: "4 jobs · 3 verified" tells you instantly
• 80% less horizontal space than the current pills
Cons:
• Loses per-vendor detail at the top level (which vendor is behind?)
• One more column than Option A (marginally wider)
• Count alone doesn't show which service is lagging

Option C: Ultra-Dense — Customer Under Trip ID, No Cargo Column

Radical
Maximize density: merge Customer as a sub-line under Trip ID (like the current design does in reverse). Drop the Cargo column (bags/weight are visible in the expanded sub-row and on the create form). This frees space for wider date and route columns.
Pros:
• Most compact option — only 7 columns total
• Trip ID + Customer in one cell = one mental unit
• Dates and route get more breathing room
• Cargo (bags/kg) rarely drives ops decisions at the trip level
Cons:
• Customer name is harder to scan vertically (tucked under ID)
• Cargo weight matters for FM trucking (truck type selection)
• Filtering by customer requires reading sub-text, not primary column

Recommendation: Option B — Trip-First + Compact Job Summary

The client's core feedback is valid: Trip ID should lead, dates should be prominent, and the current Jobs pills are too verbose. But removing job visibility entirely (Option A) sacrifices a genuinely useful signal — "is this trip healthy?" — that prevents ops from having to expand every row.

Option B gives the client what they asked for (Trip ID first, dates prominent, MAWB visible) while replacing the bloated job pills with a compact "4 · 3 verified" summary that takes ~10% of the table width instead of ~20%. The summary highlights the "worst" status (cancelled > pending > in progress > verified) so ops attention goes where it's needed.

Cargo column kept — weight/bags inform truck planning and are used in fee calculations. Option C's density gain isn't worth the loss.

Final column order: Trip → Customer → Pickup → Delivery → MAWB → Route → Cargo → Jobs (compact)
